semfs storage: pglite, Postgres, and "can grep use it?"

Why there are two kinds of "Postgres" in this project, why a real one got installed, and the one finding that decides the whole design.

One-line answer: a mount and semfs grep are two separate processes. pglite serves one connection at a time — held by the daemon — so a separate grep process can't open its own. With an external Postgres it can (tested ✓); with embedded pglite, grep must ask the daemon over IPC.

The shape of the system

semfs mounts a folder. Behind it, a daemon keeps a local cache (file metadata + content) and a semantic index (vectors). Searching is a separate command, semfs grep.

two independent OS processes ↑  both need the one index

key idea The index is not reachable through the mounted files — the filesystem shows files, not vector search. So grep must reach the index store itself. How it reaches it is the entire question below.

Three things people call "Postgres" here

Why a real Postgres got installed

We had pglite-oxide from day one — but only as a test crate, used by cargo test inside a single process. That can't cover the mount, because a mount involves two processes (daemon + grep) sharing one database. So to actually test by mounting, a real multi-connection server was needed.

unit test
cargo test --features pg-local

One process spins up pglite in-memory, connects, asserts. pglite is perfect here. This is what we "downloaded initially" — a crate, not a server.
mount test
semfs mount + semfs grep

Two processes, one shared DB. pglite (in-process) can't be shared across processes → needed brew install postgresql@17 pgvector.

The finding that decides everything

pglite-oxide serves exactly one connection at a time. Proven empirically: a 2nd concurrent connection blocks forever (a probe printed "connecting B…" and never returned). Press the buttons to see why that matters for each backend.

"One mount = one connection" — you're right

Your model holds: one mount needs one connection (the daemon's); multiple mounts = multiple independent single connections (each daemon its own pglite, or its own container namespace in a shared Postgres). Mounts never collide.

The subtlety isn't "1 vs many connections" — it's who holds the one connection. The daemon holds it for the whole mount lifetime, and grep is a different process. So the one connection is already taken when grep shows up. The clean consequence of your own model: grep shouldn't open its own connection — it should ask the daemon, the single connection's owner.

The fork

A · External Postgres

works today

You run a Postgres; semfs connects. grep opens its own connection (real Postgres = many connections). Multi-connection, multi-mount safe via the container namespace.

Cost: an external server to run.

B · Embedded pglite + grep-over-IPC

your goal · not built

semfs ships pglite. The daemon owns the one connection; grep sends its query over the daemon's existing IPC socket and gets hits back. One connection, one owner — your "single connection" model, exactly.

Cost: add a search request to the daemon IPC + make grep an IPC client for pgvector.

The matrix

User typeSQLite (default, embedded file) Embedded pglite (+ grep-over-IPC) External Postgres (shared server)
The through-line: the grep-connection finding isn't a footnote — it is the deployment fit. A self-contained single box (one sandbox, one laptop) → daemon owns the one connection, grep rides IPC → embedded (pglite or SQLite). Many processes / many machines sharing one durable index → multi-connection server → external Postgres. You pick storage by counting who needs the index, not by feature lists.
★ Two takeaways

1 · The connection model = the deployment shape. "Can grep reach the index?" is the same question as "how many processes/machines share it?" — so the single-connection finding decides storage, not a feature checklist. Embedded (single owner, IPC) for one box; external Postgres (many connections) for many.

2 · The matrix is honest, not a pitch. Per users.md's Mom-Test discipline, each cell claims fit only where the evidence points: Type 2 gains nothing from either Postgres option (SQLite already wins), and Type 1's embedded-pglite edge is real only because cold-start + self-containment are their stated priorities. No backend is sold to everyone.

Current architecture vs. the best solution

① Current — grep reaches the store itself

Two working paths today. In both, grep connects to the store directly — which is exactly why embedded pglite can't be a runtime backend.

path A · default   path B · opt-in pgvector
SQLite path shipping
mount daemon ──writes──▶ SQLite file (per-mount, on disk)
grep ──opens the file (via .semfs marker)──▶ same file

✓ two processes share a file — SQLite allows concurrent readers.
pgvector path committed
mount daemon ──conn #1──▶ External Postgres (you run it)
grep ──conn #2──▶ same server

✓ real Postgres = many connections. container-namespaced, mount-tested.
✗ The gap: embedded pglite can't be a runtime backend here — it serves one connection, the daemon holds it, and grep's own connection (the pattern both paths use) would block forever.

② Best — the daemon owns the index; grep asks it

One principle fixes the gap and unifies every persona: the daemon is the sole owner of the index connection; grep sends a query over the daemon's IPC socket and gets hits back. The backend becomes a daemon-side, pluggable choice.

★ The one-line takeaway

The best design isn't "add more backends" — it's moving who owns the connection, from grep to the daemon. That single change at once (a) unblocks embedded pglite, (b) matches the one-connection-per-mount model, and (c) turns the backend into a one-line swap that covers all three personas. The backends were never the hard part — who holds the connection was.
