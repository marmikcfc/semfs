Capturing Claude token usage — uncached, cached & completion

Why our benchmark read 5,010 when plain Claude really moved ~252,000 input tokens on the same task — and how to capture it correctly. Measured: Workspace-Bench case 289, plain ClaudeCode/Sonnet-4.6, 2026-05-27.

Level 1 · ELI5

Tokens are the words the model reads and writes

Every time the model runs, it reads a big pile of text (your instructions + the files it's looking at) and writes an answer. Each chunk of text ≈ a "token." You pay for reading and for writing.

The twist: some of what it reads is the same every turn (the standing instructions, the workspace context). Claude can remember that cheaply instead of re-reading it from scratch — that's the cache. Cheap to read again — but it's still being read. If your counter ignores "the remembered part," it thinks almost nothing happened.

Level 2 · the four buckets

One request splits into four token counts

Anthropic's usage object reports input in three pieces, plus output:

input_tokens — fresh, uncached input cache_creation_input_tokens — written to cache (first sight) cache_read_input_tokens — replayed from cache output_tokens — completion

A typical agent turn on case 289 (click the bar to see what the broken parser "sees"):

cache_read 221,569
cache_write 31,017
input
output

Full reality. The cache bands are the workspace + system context replayed every turn — cheap, but very much present.

Level 3 · the gotcha that bit us

input_tokens is the non-cached slice only

This is the trap. When prompt caching is on, the bulk of the prompt moves into the two cache fields and input_tokens collapses to a handful. A real turn from our run:

{"input_tokens": 2, "cache_creation_input_tokens": 10056,
 "cache_read_input_tokens": 0, "output_tokens": 148}

Our parser did prompt_tokens += input_tokens (counted the 2) and dropped the 10,056. It even declared cache_read/cache_write keys — then never filled them. Across the whole task:

QuantityValueNote
What the parser reported5,010input_tokens + output_tokens only
cache_read (cumulative)221,569workspace ctx replayed each turn
cache_creation (cumulative)31,017first-sight blocks
True total input≈ 252,000+uncached + cache_write + cache_read
Consequence for the benchmark: "plain claude = 5,010, codex = 143,837" was backwards. Counted properly, plain Claude's real footprint (~252K) is larger than codex's — so semfs's job (cut the context) is the right thing to measure, on a common basis.
Level 4 · don't double-count

Per-turn usage vs cumulative usage

An agent run streams many messages:

  • Each assistant message carries the usage for that turn.
  • The final result message carries the cumulative usage for the whole session (plus total_cost_usd).

So you pick one source of truth:

StrategyVerdict
Sum the final result.usage only✓ correct — it's already cumulative
Sum every assistant turn's usage✓ correct — if you do NOT also add result
Sum assistant turns and the result✗ double counts

Our parser loops over type=="result" events (one per task) → no double-count. The defect was purely which fields it read, not how it accumulated.

Watch subagents / context compaction: a long run can emit more than one result-shaped event. Prefer the SDK's final result usage, or sum the explicit four fields and trust those — not a derived total_tokens field, which (in this run) excluded cache_read and topped out at 57,832 while cache_read alone was 221,569.
Level 5 · expert — capture it right

Raw tokens vs cost, and cross-model fairness

1 · Capture the four fields verbatim. Never reduce to one int at the source.

def _accumulate(u, agg):
    agg["input"]          += u.get("input_tokens", 0)                  # uncached
    agg["cache_creation"] += u.get("cache_creation_input_tokens", 0)   # cache write (1.25x)
    agg["cache_read"]     += u.get("cache_read_input_tokens", 0)       # cache read  (0.10x)
    agg["output"]         += u.get("output_tokens", 0)                 # completion

total_input = agg["input"] + agg["cache_creation"] + agg["cache_read"]
grand_total = total_input + agg["output"]

2 · Two metrics, not one. Raw-token total answers "how much context moved." Cost answers "what it billed." Cache changes them very differently:

BucketPrice multiplier (rel. base input)
input_tokens1.0×
cache_creation_input_tokens~1.25×
cache_read_input_tokens~0.10×
output_tokens~5× (model-dependent)

3 · Cross-model comparison. Codex via OpenRouter reports the full prompt as input_tokens (no equivalent cache split surfaced); Claude via the OAuth subscription pushes the prompt into cache_read. Comparing claude's 5,010 to codex's 143,837 compares two different definitions. Normalize to "total input tokens moved" (sum all three input buckets on the Claude side) before you compare, and report a separate cost number.

4 · The fix for our pipeline — surgical, one function. In benchmarks/vendor/Workspace-Bench/evaluation/src/agents/claudecode.py, _parse_usage_from_stdout() already iterates the result events; add the two missing fields and a true total:

pt = u.get("input_tokens") or u.get("prompt_tokens") or 0
ct = u.get("output_tokens") or u.get("completion_tokens") or 0
cw = u.get("cache_creation_input_tokens") or 0
cr = u.get("cache_read_input_tokens") or 0
usage_total["prompt_tokens"]     += pt
usage_total["completion_tokens"] += ct
usage_total["cache_write"]       += cw      # was declared, never filled
usage_total["cache_read"]        += cr      # was declared, never filled
# true total input includes cache; keep components for cost weighting
usage_total["total_tokens"] = (usage_total["prompt_tokens"]
    + usage_total["cache_write"] + usage_total["cache_read"]
    + usage_total["completion_tokens"])
The data was never lost — ClaudeCode.js already writes the full msg.usage (cache fields included) into stdout. The reducer just wasn't reading them. So this is a parser-only change; no re-run of the agent is needed to re-derive past totals from saved raw/ output.
