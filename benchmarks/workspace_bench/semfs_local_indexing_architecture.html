semfs local indexing — architecture, the starvation problem, and the fix

How the local semantic index is built today, why a real workspace ends up only ~19% searchable, and the changes that fix it. Grounded in this session's runs on the PM (chanpin) workspace, case 289.

1 · Current architecture

semfs mounts a Supermemory container as a real directory. Reads are served from a local SQLite cache; a local semantic index is built over that cache so semfs grep can answer by meaning — offline.

① MOUNT — FUSE (Linux) / NFSv3 (macOS)
19 POSIX ops (lookup/read/readdir/…). Literal — no meaning. semfs grep is the one semantic verb.
▼ VfsOps
② LOCAL CACHE — SQLite (~/.cache/semfs/<org>/<tag>.db)
fs_data = verbatim file bytes (4KB chunks) — SOURCE OF TRUTH
fs_inode/fs_dentry = the tree · fs_remote = ino→doc id
chunks + vchunks(vec0) + vchunks_code + ffts(fts5) = the local semantic index (derived)
▼ on import / write (flush)
③ INDEXER — chunk → embed → store
Dual lane by file type (is_code_path):
• text → arctic-embed-s (384d) → vchunks
• code → jina-v2-base-code (768d) → vchunks_code
Embedder: LocalEmbedder{ model: Mutex<TextEmbedding> } — serialized (one chunk at a time)
▼ search / fallback
④ BACKEND — SemanticIndex trait
SqliteVecStore (local: vec0 KNN + fts5 BM25 → RRF → rerank) — default
CloudIndex (Supermemory /v4/search) — fallback when no local index
Embed source is an env dial: SEMFS_EMBED_BACKEND=local|openrouter|openai (cloud still stores vectors in this sqlite)
Key fact for everything below: the content cache (fs_data) and the semantic index (chunks/vchunks) are two separate things. semfs imports 100% of a container's files into fs_data at mount, but the index is built incrementally in the background after the mount is "ready."

3 · The problem — mount-lifecycle starvation

The index is a slow background build, but it only lives as long as the mount. The benchmark mounts → runs the agent (minutes) → unmounts. The full embed needs hours. So it gets cut off — at ~19%.

Two compounding causes:
(a) Lifecycle coupling — indexing runs only while mounted; unmount kills the rest. A 5-min agent task can't finish a multi-hour embed.
(b) Serialized embedding — Mutex<TextEmbedding> embeds one chunk at a time (~1 chunk/sec on this 4-core box), so even a long mount is slow.
Result: 4 of every 5 PM files are invisible to local semantic search. The agents passed only because case-relevant files embedded first. For a fair lite run across many cases, this is fatal — recall depends on which files happened to embed.

4 · The fix — what changes, and where

① Parallelize embedding semfs code change

Problem solved: serialized embed (cause b).
Replace the single Mutex<TextEmbedding> with a worker pool sized to available_parallelism() ("based on system info"). Embed N chunks concurrently across cores.
Effect: ~3.5 hr full PM embed → ~1 hr on this 4-core box (near-linear with cores; a GPU box → minutes). The benchmark can't do this — it's internal to the daemon (embed/local.rs + the indexer in sqlite_vec.rs).

② Persist + pre-warm once + reuse operational (benchmark)

Problem solved: lifecycle starvation (cause a), without new daemon code.
Before any agent case: semfs mount the container, leave it idle until coverage = 100%, then unmount. Chunks persist → every later case mounts a complete index instantly.
Effect: embed cost = O(1 per container × backend), not O(cases). With ① the one-time warm is ~1 hr; then 100 lite cases reuse it for free.

③ Decouple indexing from the mount (deeper) semfs code change

Problem solved: lifecycle coupling at the root (so users never hit 19% by accident).
A standalone semfs index <tag> job (or a persistent background indexer) that embeds to completion regardless of who's mounted, resumable. Plus emit startup.report() during the embed loop so the 30s readiness watchdog stays happy.
Effect: "embed once per container, reuse forever" becomes true automatically — the property the whole local-backend value rests on.

✗ Delete index on unmount already exists — but NOT for the benchmark

The daemon already has an ephemeral in-memory cache mode (daemon_runtime.rs: "nothing persists after unmount"). It's a flag, not new code.
But it's the wrong choice for benchmarking: deleting on unmount forces a full re-embed every mount → makes the 19% problem permanent and paid N times. Reserve it for the product / sandboxed-tool path (privacy, throwaway sessions), where persistence isn't wanted.
The change set, minimal: one semfs change worth doing now — ① parallelize the embed (kill the Mutex, pool sized to cores) — which speeds the ② persist + pre-warm-once operational approach. ③ is the proper long-term decoupling. Delete-on-unmount is a config, not a fix, and wrong for the lite matrix.
