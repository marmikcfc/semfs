2 · Build the semantic filesystem (4 artifacts, computed once at ingest)

① Normalized content (transcription)

each file → clean text + structured table. The HTML-xlsx becomes a real table: 
top10_product_status_table  (normalized)
rank | product                        | transaction amount | conversion rate
1    | 2015Spring Korean slim denim jacket (women)      | 37200   | 2.85%
2    | …                               | …       | …
(10 rows, ~480 tokens — NOT 30KB of HTML)
⚡ kills the openpyxl→HTMLParser fight; the source-of-truth read is now tiny & clean.

② Chunk + vector index

each file chunked & embedded; search returns top-k file:line:excerpt: 
search("best selling products …")
→ top10_…:L1-11   (0.91)  "rank|product|transaction amount|conversion rate…"
  best_seller_list:L1-9 (0.88) "1. 2015spring…37200…2.85%"
  (top-5, ~350 tokens, ranked by MEANING)
⚡ one call replaces a dozen filename-guessing finds.

③ Community digest (★ graphify)

Leiden communities + god nodes → the "what is this workspace about" map: 
WORKSPACE DIGEST (profile)
• product_data/  → best-sellers, top10, conversion  ★hub
• Sales/Orders/  → transactions
• taobao_campaigns/ → ad rules
"best-seller data → desktop/.../product_data/"
⚡ one read orients; you query the right place first try (fixes empty profile.md).

④ Typed edges (★ graphify)

relationships, confidence-tagged: 
top10_…  —contains→ products        [EXTRACTED]
best_seller_list —derived-from→ top10  [INFERRED]
product_data/ —groups→ {top10, list}  [EXTRACTED]
⚡ "what's related to best-sellers?" = one hop, no filename guessing.

2.5 · How would graphify determine the nodes here? (node & edge extraction)

A node = a meaningful "thing." For code, the language grammar defines the things → tree-sitter AST extracts functions/classes deterministically & free. This workspace is almost all data/docs — there's no grammar for "concepts," so an LLM reads each file once at ingest and names the entities (a bounded, cached call). Edges come from structure (containment, columns → EXTRACTED) and LLM-inferred relations / shared entities (INFERRED/AMBIGUOUS).

… then the graph assembles itself

            [transaction amount]      [conversion rate]          ← shared METRIC nodes
                  \          /
 best_seller_list ──[ best-selling products ]★god──  top10_table
       (.txt)    \         |  community:            /  (.xlsx)
                  \   "product sales analytics"    /
                   \____ [ product ×10 ] _________/
                              ┆ (weak, AMBIGUOUS)
        [ 02_sales_performance.json ]   [ Order_Records ]   ← other community
Files that share concept / metric / product nodes get pulled together; Leiden finds that dense cluster = the community "product sales analytics"; its most-connected node — "best-selling products" — is the god node. A query for "best selling products" lands on the hub and returns the whole relevant cluster (top10 + list), while 02_sales_performance.json / orders stay in a weakly-linked neighbor community (correctly lower-ranked, not a false hit).

Node/edge extraction is the part that determines everything — and it's the once-at-ingest cost (one bounded LLM pass per file, cached + dedup'd). Get the nodes right → every later query is a cheap lookup/traversal. Too coarse → useless; too fine → noisy — which is why graphify tags edge confidence and tunes community --resolution.
