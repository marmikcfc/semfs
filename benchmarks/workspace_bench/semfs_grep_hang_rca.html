The symptom

Every stage of the actual search succeeded (the daemon logged 65–324 hits). The grep then tried to read each hit's file off the FUSE mount to print a :line_start-line_end: — and that read hung on the CPU-starved daemon. The agent got 0 bytes.

The real root cause

CONFIRMED  The search succeeds; the semfs grep CLI hangs after it.

To format a hit as <file>:<line_start>-<line_end>:<chunk>, grep reads the hit's file off the FUSE mount:

cmd/grep.rs · read_local_or_sidecar → std::fs::read_to_string(mount.join(path)) // blocking FUSE read, NO timeout

The semfs daemon serves both the IPC search and the FUSE filesystem from contended resources (one Mutex<Connection> + the ~350% CPU load from the search itself + background L7). So a FUSE read() issued right after a heavy search blocks indefinitely → grep never finishes → codex sees nothing → reverts to manual exploration → plain-codex token count.

Neither SEARCH_DEADLINE nor SEARCH_TIMEOUT covered this — they guard the search/DB connection, not the grep's post-search file read. (why both exist → a separate note.)

The trap: a perfectly good search, sabotaged by a "nicety"
The file read existed only to compute a line range for prettier output — a cosmetic nicety. The data the agent actually needs (filepath + chunk text) was already in hand from the search. So a cosmetic, un-timed-out FUSE read was allowed to hang the entire tool, on the very mount it was searching, under the very load the search created. Classic: the failure wasn't in the hard part (retrieval) — it was in the easy part nobody guards.

The fix

DONE · tested
grep can't hang on FUSE. Line-range read is now read_file_timed — blocking read on a throwaway thread + recv_timeout (2s). On timeout → abandon the thread, trip a per-invocation circuit breaker, and print <file>:<chunk> (no line range) for every remaining hit. The whole grep pays at most one 2s budget and always returns the results.
DONE
SEARCH_DEADLINE 25s → 20s (strictly under SEARCH_TIMEOUT 25s) so the cooperative degrade reliably wins the race against the daemon's hard timeout.
DEFERRED
Decouple FUSE serving from search CPU. Bound rerank threads + throttle background L7 so a search can't starve the FUSE read() path. Ticketed (search-deadline-fails-closed-to-empty #2, parallelize-l7).

Tests: semfs 45 · semfs-core 266 green, clippy clean. New: read_file_timed_times_out_on_blocking_fifo (a writer-less FIFO blocks the read forever → the bound fires).

Investigation discipline: 3 plausible hypotheses + 1 self-inflicted parsing error, each killed with a specific piece of evidence, before the real cause emerged. The instrumented per-stage search logging is what turned "search is broken" into "search is fine, the grep hangs."
