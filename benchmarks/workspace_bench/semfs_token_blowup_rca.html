3 · Why reads are lethal in an agent loop — the multiplier

An agent re-sends the whole transcript every turn. With prompt caching that replay is cache_read. So a file read isn't paid once — it's paid once per remaining turn:

cache_read ≈ (bytes in context) × (turns they survive)

Try it: one raw-file Read

Read size: 12,000 tokens (a 47K-char HTML-xlsx)
Turns it survives in context: 22
that one Read costs 264,000 cache_read tokens

4 · Why codex shrank and Claude exploded — same tool, opposite behavior

codex — semfs REPLACED reading

profile → semfs grep → trusted the excerpt → wrote the answer.
Total tool output: 554 chars. Raw-file reads: 0.
The excerpt was the answer → never opened a file → 35K tokens.

claude — semfs ADDED to reading

semfs grep (terse) → distrusted it → Read the whole raw files (47K+36K HTML) → replayed ~22 turns.
Read output: 84,781 chars.
Paid for search and reads and extra turns → 1.23M tokens.

The principle the data reveals: in an agent loop the expensive operation is READING, not searching (reads replay ×turns in cache_read). So semantic search pays off only if it REPLACES file reading (codex). If the agent re-reads the source anyway (claude), search is pure additive overhead — you pay search + read + extra turns → worse than no search. The partial 19% index amplified it: weak hits → "verify by reading the real file" → and the real file is 40K of HTML.

5 · Is semantic grep the wrong school of thought?

No — codex proves the win is real (−76%). But the thesis must sharpen: the goal isn't better search, it's fewer bytes read. Search must make reading the source either unnecessary (the excerpt is the answer) or cheap (normalized small content) — not merely point at files. The wrong turn is letting the agent fall back to reading raw, un-normalized files.

It defends against the agent's worst instinct instead of trying to train it away — reframing semfs from "a faster grep" to "a filesystem where reading is always cheap and clean."
