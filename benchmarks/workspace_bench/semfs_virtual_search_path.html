How a virtual search path would work in semfs

Mechanism only — grounded in the real code: mount/fuse.rs (FUSE handlers), cache/profile.rs (the existing virtual-file template), backend/{mod,cloud}.rs + api/mod.rs (the search call). Pick a level, then step through the simulator.

🧭 Does it traverse a graph? No. semfs makes one POST /v4/search and gets back an already-ranked list. It does not walk, recurse, or expand nodes. Any graph/hybrid retrieval is server-side inside Supermemory — invisible to semfs. The FUSE read handler does one search() call and formats the answer. (Brute-force crawl = O(files) reads; virtual search = O(1) call + read top-k.)
Level 1 · ELI5

A magic question-folder

You make a special folder called "ask". To ask a question, you don't dig through every drawer — you just open a sticky-note named after your question, like ask/where-are-the-best-toys.

The moment you open that note, a helper elf 🧝 reads your question, runs to the smart card-catalog (one trip, not searching every shelf), and writes the 3 best answers onto the note before you finish opening it. You read the note → it lists the 3 right drawers → you open just those.

💡 The folder isn't real storage — it's a doorbell for the elf. Opening a note = ringing it. The elf does one quick lookup, not a treasure hunt.

Level 2 · Teen

A "file" that's really a question

Normally opening a file gives you bytes that were already on disk. Here, the file doesn't exist on disk — it's computed when you read it. The filename is the query:

cat  .semfs/search/best-selling-products

When you cat that, semfs notices the name lives under the special search/ folder, takes "best selling products" as a question, asks Supermemory's search once, and hands back a ranked list of real files:

best_selling_product_core_data_list.xlsx   (score 0.91)
top10_product_status_table.xlsx            (score 0.87)
problem_product_tracking.xlsx              (score 0.74)

Then you open those 3 real files normally. You never read the other 2,000 files.

💡 semfs already does this for one name — profile.md (an overview that's generated when you read it). A search path is the same trick, but the name carries a query.

Level 3 · College (CS)

Synthetic inodes + computed reads

A FUSE filesystem answers kernel callbacks. To resolve cat .semfs/search/<q>, the kernel calls, in order: lookup(".semfs") → lookup("search") → lookup("<q>") → open → read. The trick is to make those reserved names resolve to synthetic inodes that exist only in memory — never in the SQLite fs_inode table.

semfs already ships exactly this for profile.md (cache/profile.rs):

pub const PROFILE_INO:  u64 = u64::MAX - 1;   // reserved, never collides
pub const PROFILE_NAME: &str = "profile.md";
// lookup() special-cases the name → returns PROFILE_INO
// read()   on PROFILE_INO → api.get_profile() → format → bytes (cached in RwLock)

A search path reserves a small range of inodes the same way, e.g. SEARCH_DIR_INO = u64::MAX - 100, and per-query result files get inodes minted on lookup. The only handler that does real work is read(): decode the query from the path, call the semantic backend once, format the ranked hits as text, return them.

The backend call already exists — it's what semfs grep uses today (cmd/grep.rs:303):

// backend/mod.rs:23   trait SemanticIndex { async fn search(query, filepath) -> hits }
// backend/cloud.rs:34  CloudIndex::search → api.search(query, filepath)
// api/mod.rs:407       POST /v4/search  { q, containerTag, filepath? }  → ranked hits

💡 Nothing new in retrieval — we're just adding a second front door (a path) to the same search() that the grep CLI already calls. That's why it's small.

Level 4 · Developer (implementation)

Exactly what to add, and the contract

Namespace. Reserve .semfs/ at the mount root (mirror PROFILE_NAME). Under it, search/<query> where the query is percent- or +-encoded (filename rules: ≤255 bytes, no /). Optional scoping: search/<dir>/<query> maps to the filepath arg that api.search(query, filepath) already accepts.

Handlers to touch (crates/semfs-core/src/mount/fuse.rs):

Handler (line)Reserved-path behavior
lookup (188)root+.semfs → SEMFS_DIR_INO; .semfs+search → SEARCH_DIR_INO; search+<q> → mint a per-query file ino (hash(q) into a reserved high range), stash ino→query in an in-mem map
getattr (210)reserved inos → synthetic FileAttr (S_IFREG, mode 0444). Report size=0 or a cached length (see lazy note)
readdir (318)search/ → empty (or last-N queries). Do not enumerate all queries — there are infinitely many
read (425)reserved file ino → look up its query → backend.search(q, scope) → format → slice [offset..offset+size]

A SearchResults type, mirroring ProfileFile:

struct SearchResults {
    backend: Arc<dyn SemanticIndex>,
    cache: RwLock<HashMap<String, (Vec<u8>, Instant)>>,  // query → (bytes, fetched_at)
}
async fn render(&self, q: &str, scope: Option<&str>) -> Vec<u8> {
    if let Some((b, t)) = self.cache.read().get(q) {
        if t.elapsed() < TTL { return b.clone(); }      // avoid re-billing search_v4
    }
    let hits = self.backend.search(q, scope).await?;     // ONE call — no traversal
    let body = hits.iter()
        .map(|h| format!("{}\t{:.3}\t{}", h.path, h.score, h.snippet))
        .collect::<Vec<_>>().join("\n").into_bytes();
    self.cache.write().insert(q.into(), (body.clone(), Instant::now()));
    body
}

The full read, end to end:

agent: Read(".semfs/search/best+selling+products")
 └─ kernel → FUSE lookup(.semfs)→DIR_INO, lookup(search)→DIR_INO,
             lookup("best+selling+products")→file ino F  (decode+store query)
 └─ kernel → FUSE read(F, off, n)
       └─ SearchResults.render("best selling products", None)
            └─ CloudIndex::search → POST /v4/search {q, containerTag}
            └─ ranked hits [{path, score, snippet}]  ← ONE round trip
       └─ return bytes[off..off+n]
 ← agent gets ranked shortlist → then Read()s the top file paths (normal cache reads)

Contract / gotchas (important):

  • Non-idempotent reads. Two reads of the same path can differ (index moved) — so make it a read-only file (one atomic read), not a synthetic directory you traverse. Cache by (query, container_version) with a TTL so re-reads don't re-bill search_v4.
  • Lazy size. Stat-before-read means getattr happens first; either return size=0 and let clients read until EOF, or pre-compute on lookup (one search) and cache. Pre-compute is friendlier to cat.
  • Discovery still needs a hint. The path is self-describing in ls only if you list it; agents won't guess .semfs/search/<q>. The workdir hint (or MCP description) must say it exists — same requirement as semfs grep today.
  • Encoding. CJK queries (hot-selling/best-selling) and spaces must survive the filename round-trip → percent-encode; reject control chars; cap length.
  • No graph walk. render() calls search() exactly once. Re-ranking, hybrid lexical+vector, any graph expansion all happen behind /v4/search server-side. semfs stays O(1)+format.

💡 Net diff: ~one new file (cache/search.rs, ~mirrors profile.rs) + ~4 match-arms in fuse.rs. Retrieval code is untouched — you reuse the same SemanticIndex::search the grep CLI already calls.
