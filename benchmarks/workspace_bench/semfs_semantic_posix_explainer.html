Can Claude Code use semfs's semantic search through normal POSIX file ops?

Five levels — same idea, deeper each time. Pick a level, then play with the simulator below it. Grounded in the actual semfs code (FUSE handlers, profile.rs, grep.rs).

Level 1 · ELI5

The magic library with a hidden helper desk

Imagine a giant library that has a copy of every book your toy box ever had. That library is semfs — it copied the whole workspace into one room on your computer.

You ask: "Find me the books about best-selling toys." There are two ways:

  • 🐢 Walk every shelf and read every title yourself. Slow, and you get tired (this uses lots of "energy" = tokens).
  • 🧙 Ask the magic helper desk — it instantly points to the 3 right books.

Robot Claude walks every shelf 🐢 — because the helper desk is hidden behind a secret knock Claude doesn't know. Robot Codex knows the knock, so it just asks the desk 🧙 and finishes fast.

💡 The fix: put the helper desk behind a normal door with a sign — so any robot can open it the same way it opens any other door.

Level 2 · Teen

Two ways to search a folder

semfs makes a cloud memory (Supermemory) show up as normal folders on the computer. Inside, an AI agent can search two ways:

WayHowCost
🐢 Brute forceopen every file, Ctrl-F for the wordreads everything → expensive
🧙 Semanticask "what's about best sellers?" → get a ranked shortlistreads only the top hits → cheap

Right now the 🧙 option only works if you type a special grep command in the terminal. Claude Code doesn't use that — it has its own built-in search button that doesn't go through the terminal. So Claude is stuck on 🐢.

💡 Fix idea: make a special folder like .semfs/search/best-sellers. When Claude opens that "file," semfs secretly runs the smart search and hands back the answer. Now Claude's normal "open file" button = semantic search.

Level 3 · College (CS undergrad)

FUSE, syscalls, and two code paths

semfs is a FUSE filesystem: every open/read/readdir/lookup syscall on the mount is handled by the semfs daemon, which serves bytes out of a local SQLite cache (the whole 593 MB workspace lives in a fs_data BLOB table).

There are two independent paths:

PathTriggered byHits Supermemory?
POSIX read pathread/readdir/lookup (i.e. cat, ls, find, Read, Glob)No — literal cache bytes
semantic pathsemfs grep "<q>" (and a zsh grep() wrapper)Yes → search_v4

Why Claude misses it: Claude Code's Grep tool is an in-process ripgrep; it reads file bytes through FUSE and matches inside its own process. semfs never sees the query, so it can't make that search semantic. The zsh wrapper that would route a shell grep to semfs grep never fires, because Claude doesn't shell out to grep.

The template for the fix already exists: semfs serves a virtual file profile.md (profile.rs) — reading it triggers POST /v4/profile. Mirror that for search:

Read  /mnt/.semfs/search/best-sellers   →  FUSE read() sees reserved path
                                        →  backend.search("best sellers")
                                        →  returns ranked file paths + snippets

💡 Now Claude's native Read/LS/Glob reach semantic search — no shell, no special tool. Works for every agent.

Level 4 · Post-grad (systems)

Where exactly to wire it, and what breaks

The only call site of backend.search() today is crates/semfs/src/cmd/grep.rs:303 — a CLI command. The FUSE layer (mount/fuse.rs, 19 ops: lookup, getattr, readdir, open, read, write, create, mkdir, rename, symlink, …) never calls it. To make semantic reachable via POSIX we intercept a reserved path prefix in three handlers:

HandlerReserved-path behavior
lookup(parent, ".semfs")synthesize a virtual dir inode (like PROFILE_INO = u64::MAX-1)
readdir(.semfs/search/<q>)call search(q), return hits as dirents (symlinks → real files)
read(.semfs/search/<q>.md)call search(q), format ranked paths + snippets as text (mirror ProfileFile)

Design choices & failure modes:

  • Query encoding: the query lives in the path (search/best+selling+products.md). Filesystem-name limits (255 bytes, no /) cap query length — fine for keyword queries.
  • Caching: cache results per (query, container-version) with a short TTL; otherwise a re-read re-bills search_v4. profile.rs already caches in an RwLock<Option<Vec<u8>>>.
  • POSIX-correctness: readdir on a query dir is non-idempotent (results can change) and has no stable inode set — tools that stat then open may race. A read-only file per query is safer than a synthetic dir.
  • The hint is mandatory. semfs can't make Claude want the path. The workdir CLAUDE.md must say "for discovery, read .semfs/search/<query>.md." Without it, Claude crawls (observed: run 4 = 57 cache-served tool calls, 0 searches).

Route comparison:

RoutePOSIX-native?Reaches Claude tools?Effort
Virtual search path✓✓ Read/LS/Globsmall Rust (mirror profile.rs)
MCP semantic_search tool✗ new tool✓ best uptakeMCP server + harness wiring
grep() wrapper in bash env~ shell only~ Bash-tool grep onlytiny, low coverage

Why it matters (token economics): Claude's totals are ~94% cache_read — replaying the growing context each turn. Crawling grows that context (every find/Read adds files), so brute force is strictly worse than a ranked shortlist. Codex −75% came from one search_v4 + targeted reads.

Level 5 · Professor (theory)

An impedance mismatch between two namespaces

The real subject here is a layering problem: semantic retrieval is content-addressed and ranked; a filesystem is a hierarchical, stable namespace with referential transparency. Bridging them forces a choice about where the abstraction boundary sits.

1 · Why transparent interception is impossible at the read layer. Claude's Grep is ripgrep executing above the VFS boundary: it issues read(fd, buf, n) and performs matching in its own address space. The daemon observes a stream of byte requests with no query in scope — the search predicate never crosses the syscall interface. By the end-to-end argument, the function (semantic ranking) cannot be implemented in the lower layer (the FS) when the necessary information (the query intent) exists only in the upper layer (the agent). Hence semantic must be exposed as data the upper layer requests — i.e. as a name.

2 · Search-as-a-name is a synthetic filesystem. Encoding the query in the path (.semfs/search/<q>) is the Plan 9 / /proc / sysfs idiom: computation exposed through open/read. It works precisely because it moves the predicate into the namespace, where the lower layer can see it. The cost is abandoning two filesystem invariants:

  • Referential transparency — read(p) at t₀ and t₁ may differ (the index moves). A path is no longer a stable name for content; it's an RPC.
  • Idempotent enumeration — readdir of a query dir is a query, not a listing; stat-then-open can observe torn state. This is why a read-only file per query (single atomic materialization) is preferable to a synthetic directory (multi-syscall traversal with no consistency contract).

3 · The agent–daemon boundary recurs. Three candidate boundaries, each a different contract:

BoundaryContractCoupling
Shell grep wrapperlexical interception of a command namebrittle: tool must be the shell builtin; bypassed by in-process ripgrep
Synthetic VFS pathnamespace = capability; open = invokeuniversal but violates FS algebra (above)
MCP tooltyped RPC negotiated out-of-bandclean semantics, but not POSIX; requires capability discovery

Note all three still require capability discovery — the agent must know the affordance exists. No layer can infer intent it cannot observe; the hint (or MCP tools/list) is the discovery channel. This is the same reason a human needs the man page.

4 · The deeper claim. The −75% (codex) vs +8.5% (Claude) result is not a model-quality difference; it's a property of which abstraction boundary the agent's tools happen to cross. Equalizing the agents requires relocating the semantic capability to a boundary all agents traverse — and the only boundary every file tool traverses is open/read/readdir. Therefore: expose search as a synthetic, read-only, per-query file in the mount, cache it with version-keyed invalidation, and advertise it via the workdir hint. That is the minimal change that makes "semantic via POSIX" hold for an arbitrary agent.

Open question for further work: a synthetic path leaks query strings into logs/argv and into the FS namespace (injection, length limits, unicode normalization of CJK queries like hot-selling/best-selling). A typed MCP channel avoids this at the cost of POSIX-nativeness — so the "right" boundary is ultimately a function of your threat model and which agents you must support.
