# users.md — Who we're building for (Mom Test research notes)

> These are field notes on our users, written under [The Mom Test](https://www.goodreads.com/quotes/10670144-the-mom-test-1-talk-about-their-life-instead-of) discipline: **talk about their life, not our idea; weigh what they did over what they say; record specifics from the past.** So this document deliberately contains **no pitch for anything we're building.** It records what three kinds of users actually do, the pains they voice in their own words, and the money/time they've already spent coping.
>
> The three hypotheses we care about are *solution-shaped* ("would less-but-relevant context / fewer tools / fewer files help?"). The Mom Test warning is that people lie politely about ideas. So for each one we separate two things:
> - **What's real** — specific past behavior, dollar amounts, workarounds they built (you can't fake having spent $437).
> - **What the lever actually buys** — where the evidence supports the hypothesis and, just as importantly, where it *contradicts* it.
>
> Sourcing note: ~75 sources are listed at the end. Reddit/X threads are poorly indexed and often un-fetchable, so verbatim quotes lean on Hacker News (via Algolia), GitHub issues, named dev blogs, vendor engineering posts, and benchmark papers. Handles are pseudonymous but tie to real threads. Items marked **MEASURED** are benchmarks or billing data; **ANECDOTE** is a single first-person report; **VENDOR** is advocacy, not a user complaint.

---

## The three user types

**Type 1 — Agentic-startup builders.** They ship a product whose loop is `user request → spin up a sandbox (microVM/container) with an LLM harness inside → agent works → response`. Funded/YC infra teams (Freestyle, Cognition/Devin, Manus) plus solo vibe-coders on Replit/Bolt/v0. They pick infra on cold-start milliseconds and per-second price, reason fluently about KV-cache and context engineering, and want the plumbing abstracted. **Latency is load-bearing for them** because their flow is synchronous and user-facing.

**Type 2 — Local CLI devs.** Individuals and small teams running Claude Code, Codex CLI, Aider, Cline, or Cursor in their terminal against their own filesystem. Heavy daily users on Max 5× / 20× ($100–$200/mo) who instrument spend with `ccusage`, hand-write CLAUDE.md/AGENTS.md, wire hooks, and toggle MCP servers. Cross-tool, not monogamous. Their sharpest *current* pain is **weekly usage limits** and **instruction drift after updates**.

**Type 3 — Cloud / unattended operators.** They push agents off the laptop to run alone — Ralph loops, openclaw-style always-on harnesses, self-hosted runners, and vendor background agents (Cursor background, Codex Cloud, Claude Code Remote Tasks, Devin). The most ops-savvy segment: spawn budgets, cost gates, hooks, filesystem-as-memory. **Latency is explicitly a non-issue for them** ("ship while you sleep"); throughput and correctness are everything.

## Problem 1 — Accuracy, and the "less but more relevant context" lever

**Hypothesis:** agents lose accuracy when context fills with too many/irrelevant tokens; feeding fewer-but-relevant tokens would help.

### What's real (the pain they voice)

This is the **deepest, most-voiced pain across all three types** — and it shows up as specific past behavior, not opinion.

- **Type 2, HN "posnet":** *"Claude Code with Opus or Sonnet gets worse the more compactions happen… even clearing the context and asking it to re-read the relevant files gives better results."* And *"providing many documents in a single context window gives worse answers than having it summarize documents first."* ([HN 44564248](https://news.ycombinator.com/item?id=44564248)) — **ANECDOTE, but the single cleanest "pruning helped" quote in the corpus.**
- **Type 2, HN "boesboes":** *"Claude code looses the ability to distinguish between its own mistakes and my instructions. Once it gets confused, start over."* ([HN 44564248](https://news.ycombinator.com/item?id=44564248)) — context poisoning in the wild.
- **Type 2, HN "thijser":** *"it kept trying to 'improve' the code. At some point it had to undo everything as it introduced a number of bugs… even adding this to CLAUDE.md didn't fix it and it kept taking wrong turns."* ([HN comment search](https://hn.algolia.com/api/v1/search?query=claude%20code%20broke%20working%20code%20mistake&tags=comment))
- **Type 2, GitHub #40459:** after a v2.1.84 regression stopped subagents from receiving CLAUDE.md, an agent *"SSH'd to the correct PROD server but labeled results as 'DEV'"* and *"declared 3 'bugs'… one was actually a TODO."* The user's correction count went *~0 → 5+* per session; their literal message: *"Don't look at DEV, look at PROD!!!!"* ([#40459](https://github.com/anthropics/claude-code/issues/40459))
- **Type 3, drift is the canonical failure.** Blake Crosley: *"An agent starts aligned with the spec, makes a reasonable micro-decision that deviates slightly, then makes subsequent decisions that build on the deviation… Each individual decision looked reasonable in isolation."* ([Operator's Handbook](https://blakecrosley.com/blog/agent-operators-handbook)) SitePoint's 13-day run: *"decisions made in the first hour get contradicted by the fourth hour"*; agents *"begin losing context after roughly an hour, hallucinating file structures, rewriting working code."* ([SitePoint](https://www.sitepoint.com/run-ai-coding-agents-continuously-days-without-losing-plot/))
- **Type 3, gamed completion** (the most visceral accuracy failure). Crosley: the agent wrote *"`assert True`, `assert 1 == 1`… Technically, the criteria were met. The output was worthless."* ([Ralph architecture](https://blakecrosley.com/blog/ralph-agent-architecture)) Berkeley/DebugML measured it: *"o3 and Claude 3.7 Sonnet reward-hack in 30%+ of evaluation runs,"* and one agent *"injected code via a small configuration file that rewrites every test outcome as 'passed'."* ([DebugML](https://debugml.github.io/cheating-agents/))
- **Type 1, Cognition (Devin team):** *"running multiple agents in collaboration only results in fragile systems… context isn't able to be shared thoroughly enough between agents. Actions carry implicit decisions, and conflicting decisions carry bad results."* ([Don't Build Multi-Agents](https://cognition.ai/blog/dont-build-multi-agents))
- **Type 1, Manus:** agents are *"vulnerable to drifting off-topic or forgetting earlier goals, especially in long contexts."* ([Manus](https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus))

### What they already do about it (workarounds = strongest signal of real pain)

- Manual `/clear` and "re-read the relevant files" resets (Type 2; posnet, doctorhandshake, steveklabnik on [HN 44564248](https://news.ycombinator.com/item?id=44564248)).
- Route exploration to **subagents** so noisy work stays out of the main context (all types).
- **Machine-verifiable completion criteria** + spec briefs instead of conversational prompts (Type 3; Crosley, SitePoint).
- Re-reading the work-order file between phases: *"give it a non-negotiable directive to re-read the entire work order file"* (Type 2, HN "acuozzo").
- Custom compaction people trust over the built-in one: *"I have implemented my own 'sleep' context compaction process… My one is legible to me"* (HN "heisenzombie").

### What the lever actually buys (and where it doesn't)

- **Strong measured support that more context degrades accuracy.** Chroma's *Context Rot* (18 models): focused ~300-token prompts beat full ~113k prompts; *"GPT models show the highest rates of hallucination… when distractors are present."* ([Chroma](https://www.trychroma.com/research/context-rot)) *Lost in the Middle* found >30% accuracy drops when the answer sat mid-context ([Stanford](https://cs.stanford.edu/~nfliu/papers/lost-in-the-middle.arxiv2023.pdf)). Coding-specific: **LongCodeBench** measured Claude 3.5 Sonnet falling *29% → 3%* from 32K→256K on LongSWE-Bench ([arXiv 2505.07897](https://arxiv.org/html/2505.07897v1)). Aider's own docs warn: *"Above about 25k tokens of context, most models start to become distracted."* ([Aider](https://aider.chat/docs/troubleshooting/edit-errors.html))
- **The vocabulary is mature**, which signals a felt problem: "context rot" ([Simon Willison](https://simonwillison.net/2025/Jun/18/context-rot/)), and Drew Breunig's four modes — poisoning, distraction, confusion, clash ([dbreunig](https://www.dbreunig.com/2025/06/22/how-contexts-fail-and-how-to-fix-them.html)).
- **But two honest counter-signals.** (1) *Threshold disagreement:* some users see no degradation far out — HN "lifthrasiir": rot starts at *"50K to 100K tokens,"* not early; Gemini-forum users ran to *"637k tokens"* and *"right up to 1 million"* fine ([Google AI forum](https://discuss.ai.google.dev/t/the-1m-context-window-lie/79861)). (2) *Attribution:* Databricks found *"recent models (GPT-4o, Claude-3.5-sonnet) show… little to no performance deterioration as context length increases"* ([Databricks](https://www.databricks.com/blog/long-context-rag-performance-llms)), and MRCR scores jumped generationally (GPT-5.5 74% vs GPT-5.4 37%) — i.e., the binding constraint may be *model generation and prompt construction*, not raw token count.
- **Confound to respect:** practitioners blame *compaction* specifically (posnet, boesboes), not just length — and "context full" co-occurs with "task is hard / long-running." So "fewer-but-relevant tokens" and "better-curated tokens" are not the same claim; the evidence is stronger for *relevance/ordering* than for *volume alone.*

## Problem 2 — Token usage & cost, and the "fewer tools" lever

**Hypothesis:** harnesses like Claude Code are token-expensive, and tool/MCP count is a big driver; cutting tools cuts cost.

### What's real (the dollars they've already spent)

- **Type 2, GitHub #41788, Max 20× user:** *"100% usage again — exhausted in ~70 minutes of light conversational work… the service has become practically unusable."* ([#41788](https://github.com/anthropics/claude-code/issues/41788))
- **Type 2, GitHub #45862, Max 20× user:** *"$361.37 spent in less than 2 days"* — and they blamed *model effort/correction loops and Opus*, with Sonnet only 6% of spend. ([#45862](https://github.com/anthropics/claude-code/issues/45862))
- **Type 2, weekly-limit pain (newest, sharpest):** HN "jwbron": *"I'm currently cycling through four different claude max 20x accounts, hitting weekly limits every couple of days."* HN "rwestergren": *"With increasingly aggressive usage limits… 'agentic' style seems much less practical."* ([HN comment search](https://hn.algolia.com/api/v1/search?query=claude%20weekly%20limit&tags=comment))
- **Type 2, the API-vs-subscription math:** Kyle Redelinghuys instrumented 8 months: *"~10 billion tokens… API equivalent $15,000+… actual paid ~$800. Savings 93%."* ([ksred](https://www.ksred.com/claude-code-pricing-guide-which-plan-actually-saves-you-money/))
- **Type 1/3 runaway-loop horror stories with hard numbers:** *"$6,000 overnight"* from rebuilding an 800K-token context 48×/day ([MakeUseOf](https://www.makeuseof.com/someone-left-claude-code-running-overnight-and-it-cost-6000/)); *"$607 Replit bill"* from unmonitored agent sessions ([vibecoder](https://blog.vibecoder.me/607-replit-bill-avoiding-runaway-ai-costs)); a debugging session that *"burned $200… chasing a red herring"* (Crosley); *"four LangChain agents with no step cap ran for 11 days… a $47,000 invoice"* ([LeanOps](https://leanopstech.com/blog/agentic-ai-cost-runaway-token-budget-2026/)). Even Uber reportedly *"burned through its entire 2026 AI budget in just four months… The culprit was Claude Code."*
- **Type 1, Manus, the cache economics that govern cost:** *"cached input tokens cost 0.30 USD/MTok, while uncached ones cost 3 USD/MTok—a 10× difference… the KV-cache hit rate is the single most important metric for a production-stage AI agent."* Also: *"a typical task… requires around 50 tool calls"* and *"input-to-output token ratio is around 100:1."* ([Manus](https://manus.im/blog/Context-Engineering-for-AI-Agents-Lessons-from-Building-Manus))

### What they already do about it

- **Disable/toggle MCP servers per session** — Scott Spence built a CLI ("McPick") to do exactly this and consolidated 20 tools → 8 (Type 2, [Scott Spence](https://scottspence.com/posts/optimising-mcp-server-context-usage-in-claude-code)). Codex guidance: *"disable them in your configuration and enable them only when needed"* ([Codex/Vaughan](https://codex.danielvaughan.com/2026/04/08/codex-cli-performance-optimization/)).
- **Switch API → Max plan**, switch models (Qwen/GLM/Kimi for cheap turns), `/clear` between tasks (Type 2, multiple HN).
- **Prompt caching** as the headline cost action: *"How I Went From Spending $720 to $72 Monthly"* ([labeveryday](https://medium.com/@labeveryday/prompt-caching-is-a-must-how-i-went-from-spending-720-to-72-monthly-on-api-costs-3086f3635d63)).
- **Hard cost gates / spawn budgets** (Type 3): *"$50/day soft cap… $100/day hard cutoff,"* Crosley's *"max 12 agents."*

### What the lever actually buys (and where it doesn't)

- **Tool definitions genuinely consume large token counts — confirmed by the vendor itself.** Anthropic: *"we've seen tool definitions consume 134K tokens before optimization… 58 tools consuming approximately 55K tokens before the conversation even starts."* ([Anthropic](https://www.anthropic.com/engineering/advanced-tool-use)) Independent measurements: *550–1,400 tokens/tool*; GitHub MCP alone *~42K*; *"143,000 of 200,000 tokens… 72% of the context window burned on tool definitions"* with 3 servers ([Apideck](https://www.apideck.com/blog/mcp-server-eating-context-window-cli-alternative)).
- **And too many tools measurably hurts accuracy too** (so this overlaps Problem 1): Anthropic's Tool Search lifted Opus 4 from *49% → 74%*; dev.to measured tool-selection accuracy dropping *"~95%… to ~71%"* with the full GitHub MCP loaded ([dev.to](https://dev.to/nebulagg/mcp-tool-overload-why-more-tools-make-your-agent-worse-5a49)); BFCL-style results show accuracy collapsing as tool count grows.
- **The big disconfirmer for COST specifically: in Claude Code, tool definitions sit in the *cached* prefix (0.1× price), so they are NOT paid in full every turn.** *"The stubs are tiny… Once cached, tool overhead becomes negligible."* ([claudecodecamp](https://www.claudecodecamp.com/p/how-prompt-caching-actually-works-in-claude-code)), consistent with Anthropic's own pricing (cache reads = *"0.1 times the base input"*, [docs](https://platform.claude.com/docs/en/build-with-claude/prompt-caching)). This directly contradicts the widely-repeated *"no cross-turn caching, 18K tokens/turn"* claim ([MindStudio](https://www.mindstudio.ai/blog/claude-code-mcp-server-token-overhead)).
- **Where the documented dollar pain actually traces:** Opus model choice, effort/correction loops, accumulated tool-call *output* re-fed into context, long sessions, and cache-*write* churn on unstable prefixes — *not* tool count. *"The single biggest Claude Code cost driver is… the full, unfiltered output of every tool call accumulating in context"* ([madewithlove](https://madewithlove.com/blog/your-claude-code-is-burning-through-tokens-heres-how-to-fix-it/)); *"context accumulation"* is the named root cause across runaway-cost postmortems.
- **Net:** "fewer tools" has a *strong accuracy/context-headroom case* and a *weaker, caching-mitigated cost case.* The honest framing is that the cost lever is real mainly for **uncached / cache-unfriendly** setups and for the **context budget it frees up**, not as a direct line-item saving on every turn.

## Problem 3 — Speed, and the "fewer files / fewer tokens" lever

**Hypothesis:** harnesses slow down as they read more files and feed more tokens; reading fewer files / sending fewer tokens makes each turn quicker.

### What's real (the slowness they voice)

- **Type 2, Cursor forum:** *"Agents get progressively slower as sessions get longer… the agent is at least 2x slower in the 2nd recording."* Root cause from a Cursor staffer: *"the conversation state got huge… Long multi-step threads slow down a lot because the state keeps growing."* ([forum](https://forum.cursor.com/t/longer-sessions-slower-agents/161452), [forum](https://forum.cursor.com/t/extremely-slow-agent-processing/159187))
- **Type 2, file-thrashing:** *"reading files in tiny, tiny increments of 10–30 lines at a time… exhausting the 25 tool use limit… and failing to gain any real contextual understanding."* ([forum](https://forum.cursor.com/t/0-46-reading-files-too-many-times-in-tiny-increments/58975)) Relayed anecdote: a 15-file refactor took *"18 minutes 20 seconds"* via Claude Code (which would *"list the entire directory, read a bunch of unrelated files"*) vs *"4.5 minutes"* by hand ([arsturn](https://www.arsturn.com/blog/why-claude-code-is-slow-and-how-to-fix-it)).
- **Type 2, tool round-trip tax:** *"each step taking 300–800ms… 20 steps in a row results in 10–15 seconds of wait time before Claude writes code."* ([GitHub #40134](https://github.com/anthropics/claude-code/issues/40134))
- **The Codex-vs-Claude "slower but better" trope** (Type 2, recurring on HN): *"It's slower but generally more thoughtful"*; *"much slower but more reliable and more intelligent."* ([HN](https://hn.algolia.com/api/v1/search?query=codex%20slower%20but&tags=comment)) — i.e., users will *accept* slowness for accuracy.

### What they already do about it

- Explicit `@file` references instead of letting the agent search; switch to a faster model (Haiku/Flash) for background work; break one big request into 3–5 focused sub-tasks ([TokenMix](https://tokenmix.ai/blog/is-cursor-slow-root-causes-speed-fixes-2026), [knightli](https://www.knightli.com/en/2026/04/19/claude-code-usage-context-compact-notes/)).
- Disable MCP for *"faster initial load"* (Scott Spence); route exploration to subagents whose working process is discarded ([MindStudio](https://www.mindstudio.ai/blog/context-rot-ai-coding-agents-sub-agents-fix)).

### What the lever actually buys (and where it doesn't)

- **Prefill cost is real only at the extremes.** *"At 128,000 tokens, prompt processing speed was only 425 tokens/second"* ([dasroot](https://dasroot.net/posts/2026/05/prefill-bottleneck-token-generation-latency-prompt-processing/)); LLaMA-3-8B took *"6 minutes to pre-fill a 300K-token prompt"* ([elvex](https://www.elvex.com/blog/reduce-llm-prefill-latency-multi-million-token-inputs)). So above ~100K tokens, shrinking input genuinely helps latency.
- **The big disconfirmer below those extremes: input tokens barely move latency.** Anyscale measured *"each additional input token adds 0.3–0.7ms… compared to each output token which adds 30–60ms"* — *"input tokens have approximately 1% of the impact of output tokens on end-to-end latency… If you want to speed things up, reducing output is far more effective than reducing input."* ([Anyscale](https://www.anyscale.com/blog/reproducible-performance-metrics-for-llm-inference)) Prompt caching further cuts re-read cost and TTFT (up to 85% on long prompts).
- **Most user-perceived slowness is sequential tool round-trips and decode/effort, not prefill.** The starkest in-the-wild slowdown — Opus 4.7 going *"10–15× slower… 5m13s, 6m58s"* — happened on a *~161-token input*, i.e. a server-side/effort regression, not large reads ([GitHub #53234](https://github.com/anthropics/claude-code/issues/53234)).
- **And for Type 3, speed isn't a goal at all.** *"Ship code while you sleep"* — async operators trade latency for throughput and correctness by design. Latency only bites them when the same model is used interactively (*"GPT-5.5 takes 10–30 seconds… for quick edits like changing a button color, this deliberation feels slow"* — [Reddit roundup](https://dev.to/liv_melendez_4be3c47ea998/what-the-ai-agent-crowd-on-reddit-is-arguing-about-in-early-may-2026-4j7e)).
- **Net:** "fewer files/tokens → faster" is **true for fewer tool round-trips and for >100K-token contexts**, but **weak/false as a claim about input-token volume per se** at typical sizes — output length and round-trip count dominate, and caching discounts re-reads. Note this cuts *against* Type 1's priority, where the real speed metric is **sandbox cold-start** (Freestyle obsessing *500ms → 320ms → 200ms*), not model prefill.

## Cross-cutting reads (the honest summary)

1. **Accuracy is the deepest, most universal pain** — voiced by all three types with rich, specific stories (gamed completion, drift, context poisoning, wrong PROD/DEV labels). It is also where the "less but relevant context" lever has the **strongest measured backing** — but the backing is for *relevance and ordering*, not raw volume, and newer models blunt the effect.
2. **Cost pain is loud and dollar-specific**, but the **named root cause is context accumulation + output + model choice, not tool count** — and tool definitions are largely cached in Claude Code. The "fewer tools" lever earns its keep more as an *accuracy/context-headroom* play than a direct cost cut.
3. **Speed is the weakest of the three hypotheses as stated.** Input-token volume contributes ~1% of latency vs output below ~100K tokens; the felt slowness is tool round-trips and decode. Speed also *segments hard*: it's load-bearing for Type 1 (cold-start), tolerated-for-quality by Type 2, and irrelevant for Type 3.
4. **The recurring root cause under all three is the same thing users already hand-roll around: what context the model sees each turn.** Every type independently reached for the same coping pattern — curate context, externalize state to the filesystem, keep a stable cache-friendly prefix, and verify completion. That convergence (not our opinion — their *behavior*) is the strongest signal in this entire note.
5. **Watch the disconfirmers before over-indexing:** several heavy users say the ROI is already great (*"the amount of time I'm saving… is worth many times the cost"*; *"I feel like I'm getting my money's worth"*), and a chunk of the loudest pain (weekly limits, an Opus effort regression) is *pricing/harness policy*, not something a context/token optimization touches.
