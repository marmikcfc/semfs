The caretaker's footprint: four little files

One mount = one tag = one daemon (🧑‍🔧). That daemon drops exactly four files on disk. Learn these and you can debug almost anything.

<cache_dir>/ Everything one mount needs lives under one folder
<tag>.db 🗄️ The Cache — a SQLite database holding your bytes + the write queue
sockets/<tag>.sock The Unix socket — the phone line the CLI uses to talk to the daemon
pids/<tag>.pid The daemon's process ID — its “on duty” badge number
logs/<tag>.log The logbook — your first stop when something is wrong
semfs status Reads the pid file and tells you if the caretaker (🧑‍🔧) is alive
semfs logs Prints the logbook for that tag — what the daemon has been doing

“Is the caretaker alive?” in real code

Here is the exact function semfs uses to check whether the daemon is still running. The trick is delightfully simple — and a little scary-sounding.

CODE 
/// Is the given pid alive? POSIX: shells out to `kill -0 <pid>`, which
/// does not actually send a signal — it's just a liveness probe that
/// succeeds iff the process exists and we're permitted to signal it.
pub fn pid_alive(pid: u32) -> bool {
    std::process::Command::new("kill")
        .arg("-0")
        .arg(pid.to_string())
        .stdout(std::process::Stdio::null())
        .stderr(std::process::Stdio::null())
        .status()
        .map(|s| s.success())
        .unwrap_or(false)
}
PLAIN ENGLISH

The comment is the key: this is just a “is it alive?” probe.

Despite the name, kill -0 does NOT kill anything.

It just asks the OS “does a process with this badge number exist?”

Define a function that takes a signal-free check.

Run the real kill program as a tiny helper…

…with the -0 flag (the “just check” mode)…

…pointed at the daemon's badge number.

Throw away whatever it prints — we only want the verdict.

Same for error text — silence it.

Actually run it and grab the result.

Did it succeed? Then yes, the daemon is alive.

If anything went wrong, assume it is dead (safe default).

Done.

🛡️
Why a subprocess instead of doing it directly?

The whole crate forbids unsafe code, so semfs borrows the operating system's own kill tool rather than poking the kernel itself. Clean and honest.

The “out of order” sign nobody cleared

Your data was never at risk

The Cache (🗄️) is a SQLite database with a WAL, so a crash cannot corrupt it. Queued writes simply resume and the SyncEngine (🚚) drives them to the Backend (📚).

The Operator's Debug Routine

1
semfs status — is the daemon alive?

This runs pid_alive for you. Dead caretaker? That explains a hung mount instantly.

2
semfs logs — what does the logbook say?

Read the per-tag log (🗄️). Almost every mystery confesses itself here.

3
Read the INFO / WARN / STOP tiers — patient or stuck?

When the Backend (📚) is slowly processing a file, semfs escalates its messages. INFO = normal. WARN = taking a while. STOP = genuinely stuck. This is how you know whether to wait or worry.

4
Stale files after a crash? Cleanup runs on next mount.

cleanup_stale clears the leftover sign automatically — usually nothing for you to do.

5
Still stuck? semfs unmount --force

A force unmount tears down a wedged mount so you can start fresh.

6
Re-mount with the firehose on

Set the environment variable RUST_LOG to crank up log detail: RUST_LOG=semfs=debug,semfs_core=trace semfs mount …
