The everyday moment

echo "remember the deploy runs migrations first" > ./notes/deploy.md — done before you blink. But where did your words actually go?

Meet semfs — short for semantic filesystem. It is a single self-contained binary that makes a storage service behave like an ordinary filesystem folder.

The whole point: every tool you already own — cat, ls, grep, your editor, git — already speaks "files." So memory delivered as files needs no new SDK and no special tool.

The Dropbox at the Post Office

Picture sliding a letter through the slot of a post-office dropbox. The instant it drops, it is safe — the post office owns it now, it will not blow away. But it has not been delivered yet.

✉️
The letter drops — it is safe NOW

Your write lands locally the instant the prompt returns. It will survive even if you pull the plug.

🚚
A courier delivers it LATER

Sending it on to the real destination happens afterward, quietly, in the background.

💡
The one idea that explains everything

A write is durable the instant it lands locally, but delivery to the real backend happens later, in the background. This explains every "my note did not show up on the other machine yet" question you will ever have.

The Whole Journey, In Four Lines

Here is a real session from the project README. Read the left as code, the right as plain English.

CODE 
# 2. mount a memory container as a folder
semfs mount my-notes --path ./my-notes

# 3. use it like any folder
echo "the deploy pipeline runs migrations before swapping traffic" > ./my-notes/deploy.md
cat ./my-notes/deploy.md

# 4. search by meaning (run from inside the mount)
cd ./my-notes && semfs grep "how are schema changes applied during a release"
#   → deploy.md:1:the deploy pipeline runs migrations before swapping traffic
PLAIN ENGLISH

Attach a memory container named "my-notes" so it appears as a real folder.

It now lives at ./my-notes — open it like any folder.

 

Write a sentence into a file. The prompt returns instantly — because the write only had to reach your local disk, not the cloud.

Read it straight back. You get the exact bytes you wrote.

 

Step into the folder and ask a question in plain language.

The answer: the matching file, line number, and the verbatim text.

🔎
Look closely at the magic

Your question — "schema changes during a release" — shares almost no words with the file — "deploy pipeline runs migrations." Yet it matched. That is semantic search, the killer feature.

Why grep-by-meaning Is a Superpower

Here is the output format you get back when you ask a question:

CODE 
$ semfs grep "credential renewal flow"
auth.md:12:the access token is refreshed by the middleware before each request
PLAIN ENGLISH

You ask about "credential renewal" — words that never appear in the file.

It still finds the right line and hands back exactly three things: filepath : line : chunk. The chunk is verbatim text from the file.

Behind the scenes, the backend turns each line into an embedding and builds an index over them. That is what lets it match by idea, not by letters.

🤖
Why this matters for AI agents

An AI agent can grab just the relevant lines instead of stuffing whole files into its limited context window. Memory-as-files gives the model a stable surface it already knows how to use.
