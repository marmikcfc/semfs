The Outside World: Backends and the Couriers That Reach Them

Back in module 1, your write was "durable the moment the prompt returned" — but it had not reached the cloud yet. So who actually drives it there? And what happens if the cloud is slow, offline, or a teammate deleted a file on another machine? Meet the courier fleet.

A shipping logistics network

Your cache 🗄️ is the warehouse next door: instant. The backend 📚 is a distant fulfillment center. Between them runs a fleet of couriers — the SyncEngine 🚚 — each with its own route and schedule, so YOU never wait at the loading dock.

semfs offers three backend tiers. The filesystem behaves identically across all of them — graceful degradation, not lowest common denominator.

📦

Embedded (the default)

SQLite + sqlite-vec + FTS5. Local, offline, one single binary.

Best for: a personal mount on one machine.

🏢

Server

Postgres + pgvector. Concurrent multi-writer, large corpora (HNSW index).

Best for: a team, many documents.

☁️

Cloud

Supermemory API. Zero local index — the server keeps a memory graph for you.

Best for: shared, managed memory you do not host yourself.

The dispatcher has four dials

The SyncEngine reads its schedule from one small settings struct. Two of the dials are timers; two are simple on/off switches that the mount flags flip.

CODE 
/// Knobs for the sync engine. All optional — defaults are production-sane.
#[derive(Debug, Clone, Copy)]
pub struct SyncOptions {
    pub delta_interval: Duration,
    pub deletion_scan_interval: Duration,
    pub pull_enabled: bool,
    pub push_enabled: bool,
}

impl Default for SyncOptions {
    fn default() -> Self {
        Self {
            delta_interval: Duration::from_secs(30),
            deletion_scan_interval: Duration::from_secs(300),
            pull_enabled: true,
            push_enabled: true,
        }
    }
}
PLAIN ENGLISH

These are the dials. Every one is optional — the defaults are sensible for real use.

There are exactly four of them.

delta_interval: how often to check the backend for newer documents.

deletion_scan_interval: how often to check what got deleted remotely.

pull_enabled: master switch for bringing changes IN. --no-sync flips this off.

push_enabled: master switch for sending changes OUT. --no-push flips this off.

The defaults: pull every 30 seconds...

...scan for deletions every 300 seconds (5 minutes)...

...and both switches start ON, so a fresh mount syncs both ways.

🎛️
A mount flag is not magic

A flag just decides which couriers clock in. --no-push means your writes never leave your machine — exactly how you read a shared container without contaminating it.

The five couriers and their schedules

With both switches on, five background loops run on their own timers. Each has one job.

🚚
Loop A — delta pull (every ~30s)

Fetches documents sorted newest-first and reconciles anything newer than our watermark into the cache.

🗑️
Loop C — deletion scan (every ~5min)

Compares all remote IDs against local records and unlinks anything that vanished from the backend.

📤
Loop D — push worker

Sends queued local writes up to the backend. Coalesces rapid edits to at most 2 requests per file. (Deep dive in module 5.)

⏳
Loop E — inflight poller

Watches documents the server is still processing (extract → chunk → embed → index → done) and updates their status by polling.

💧
Loop F — hydration worker

Pulls the actual file contents on demand, only when you open something.

💡
Durable now, delivered later

semfs splits durable (instant, local) from delivered (eventual, remote). Engineers call this write-behind caching — speed and safety without making you wait on the network.

The async-processing trap

Send a second PATCH while the first is still processing and the server silently drops your new content. So semfs waits for status=done before sending the next write on the same file — which is why edits can take a moment to appear in search.
