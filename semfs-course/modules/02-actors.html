Meet the Cast

Last module you watched your words pass through four hands on the way from your keyboard to the cloud. Those hands have names. Let's meet the crew.

One seamless show, a whole crew backstage

From the outside, semfs looks like a single thing: a folder you can read and write. Behind the curtain, a small crew runs the show — and each member has exactly one job.

🎭
The big idea

semfs is built from a handful of components, each with a single responsibility, talking through clean boundaries. Engineers call this separation of concerns. It's what lets you tell an AI “the bug is in the sync layer, not the mount layer” instead of “it's broken somewhere.”

🎫
The CLI — the front desk

Parses your typed command like semfs mount and hands off. It's a thin CLI dispatch layer.

🚪
The Mount adapter — the translator at the door

Turns kernel filesystem calls into plain method calls. The only place that knows about FUSE or NFS.

📜
The VFS / FileSystem trait — the contract everyone signs

Defines what any filesystem must be able to do (lookup, read, write, mkdir). It's the rulebook — we go deep on this trait next module.

🗄️
The Cache / CacheFs — the fast local vault

A SQLite database that stores your bytes durably and instantly, right on your machine.

🚚
The SyncEngine — the courier

Background async loops that carry your writes up to the backend and pull remote changes back down.

📚
The Backend / SemanticIndex — the librarian

Answers grep by meaning, not just exact words. Can be cloud, Postgres, or local.

🧑‍🔧
The Daemon — the stage manager

The long-running daemon process that owns an active mount, with its socket, PID, and log files.

Notice the handoff

Your prompt comes back the instant the Cache says “durable” — long before the SyncEngine and Backend finish their slow background work. That split is why writes feel instant.

The cast list, in the code's own words

Open the library's table of contents and you'll find the entire crew listed as one module each.

CODE 
pub mod agent_hint;
pub mod api;
pub mod backend;
pub mod cache;
pub mod config;
pub mod daemon;
pub mod mount;
pub mod sync;
pub mod vfs;
PLAIN ENGLISH

Each pub mod line is one crew member, declared once.

backend is the librarian who answers grep by meaning.

cache is the fast local vault that stores your bytes.

daemon is the stage manager that owns the mount.

mount is the door that translates kernel calls.

sync is the courier carrying changes up and down.

vfs is the contract — the rulebook everyone signs.

And the runnable program? Almost empty. The binary just parses and dispatches:

CODE 
#[tokio::main]
async fn main() -> Result<()> {
    init_tracing();
    let cli = Cli::parse();
    cmd::dispatch(cli.command).await
}
PLAIN ENGLISH

Turn on logging so we can see what happened.

Read the command the user typed.

Hand it to the right handler — and that's it.

All the interesting code lives in the library, not here.

Only the door knows about FUSE and NFS

There are two ways to make the OS see a folder. The Mount adapter hides that choice from everyone else — so the rest of the crew never has to care which one is in play.

CODE 
pub enum MountBackend {
    /// FUSE — Linux only. Uses the `fuser` crate.
    Fuse,

    /// NFSv3 over localhost — works on both macOS and Linux.
    Nfs,
}
PLAIN ENGLISH

Two doors to choose from, listed in one place.

On Linux, use FUSE to back the folder.

On macOS (and Linux), use a tiny local NFS server instead.

Everything else in semfs is blissfully unaware which door is open — that's the whole point of separating this out.
