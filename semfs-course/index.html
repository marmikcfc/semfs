Where Did Your Words Go?

You type this into a folder, hit enter, and the prompt comes back instantly:

⌨️
The everyday moment

echo "remember the deploy runs migrations first" > ./notes/deploy.md — done before you blink. But where did your words actually go?

Meet semfs — short for semantic filesystem. It is a single self-contained binary that makes a storage service behave like an ordinary filesystem folder.

The whole point: every tool you already own — cat, ls, grep, your editor, git — already speaks "files." So memory delivered as files needs no new SDK and no special tool.

The Dropbox at the Post Office

Picture sliding a letter through the slot of a post-office dropbox. The instant it drops, it is safe — the post office owns it now, it will not blow away. But it has not been delivered yet.

✉️
The letter drops — it is safe NOW

Your write lands locally the instant the prompt returns. It will survive even if you pull the plug.

🚚
A courier delivers it LATER

Sending it on to the real destination happens afterward, quietly, in the background.

💡
The one idea that explains everything

A write is durable the instant it lands locally, but delivery to the real backend happens later, in the background. This explains every "my note did not show up on the other machine yet" question you will ever have.

Meet the Cast

Last module you watched your words pass through four hands on the way from your keyboard to the cloud. Those hands have names. Let's meet the crew.

One seamless show, a whole crew backstage

From the outside, semfs looks like a single thing: a folder you can read and write. Behind the curtain, a small crew runs the show — and each member has exactly one job.

🎭
The big idea

semfs is built from a handful of components, each with a single responsibility, talking through clean boundaries. Engineers call this separation of concerns. It's what lets you tell an AI “the bug is in the sync layer, not the mount layer” instead of “it's broken somewhere.”

🎫
The CLI — the front desk

Parses your typed command like semfs mount and hands off. It's a thin CLI dispatch layer.

🚪
The Mount adapter — the translator at the door

Turns kernel filesystem calls into plain method calls. The only place that knows about FUSE or NFS.

📜
The VFS / FileSystem trait — the contract everyone signs

Defines what any filesystem must be able to do (lookup, read, write, mkdir). It's the rulebook — we go deep on this trait next module.

🗄️
The Cache / CacheFs — the fast local vault

A SQLite database that stores your bytes durably and instantly, right on your machine.

🚚
The SyncEngine — the courier

Background async loops that carry your writes up to the backend and pull remote changes back down.

📚
The Backend / SemanticIndex — the librarian

Answers grep by meaning, not just exact words. Can be cloud, Postgres, or local.

🧑‍🔧
The Daemon — the stage manager

The long-running daemon process that owns an active mount, with its socket, PID, and log files.

The Trait Seam: One Folder, Any Storage

semfs can store your memory in a local file, a database, or a cloud service — and the folder behaves identically across all three. The trick is a single idea you already use every day: the power socket.

The wall socket nobody thinks about

🔌
Your charger plugs into a standard shape

It does not care if the power comes from a coal plant, a solar farm, or a nuclear reactor.

⚡
The socket is a contract

As long as 120V arrives through that shape, you can swap the entire power plant behind the wall.

🧩
A trait is that socket

In semfs code, a trait is a standard shape that many different storage engines can plug into.

💡
The single most useful sentence you can say to an AI

This is called programming to an interface, not an implementation. It is THE most transferable idea in this whole course.

Where the two halves meet

semfs defines what a filesystem must do separately from how any storage does it. The two halves meet at one contract — engineers call that meeting point the seam.

One shape, fulfilled three ways

Same method name — different machinery behind it. Flip between the contract and its two backends.

trait FileSystem The socket. It declares the method lookup but writes zero code for it — it only states that the ability must exist.
fn lookup(...) "Find a file by name inside a folder." WHAT, never HOW.
MemFs In-memory backend used in tests. Fast, disposable, no disk.
fn lookup(...) Fulfills the same method by scanning a table held in RAM. Same shape, RAM machinery.
CacheFs The real backend, backed by a SQLite file on disk.
fn lookup(...) Fulfills the same method by running a SQL query against the database. Same shape, disk machinery.

The contract names the ability. Each backend supplies its own machinery. Calling code never sees the difference.

Proof: an 8-line fake fools the search

The search socket is a trait called SemanticIndex. The real cloud plugs into it — but so can a tiny fake used in tests. Search cannot tell them apart.

CODE 
/// The semantic-search substrate behind `grep`. Any backend that answers a semantic query.
#[async_trait]
pub trait SemanticIndex: Send + Sync {
    /// Search by meaning. `filepath` optionally scopes to a prefix.
    async fn search(&self, query: &str, filepath: Option<&str>)
        -> anyhow::Result<Vec<SearchHit>>;
}
 
    struct FakeIndex;
 
    #[async_trait]
    impl SemanticIndex for FakeIndex {
        async fn search(&self, query: &str, _filepath: Option<&str>)
            -> anyhow::Result<Vec<SearchHit>> {
            Ok(vec![SearchHit {
                filepath: Some("/notes/a.md".into()),
                memory: None,
                chunk: Some(format!("matched: {query}")),
                similarity: 0.9,
            }])
        }
    }
PLAIN ENGLISH

This trait is the search socket. Anything that answers a meaning-based query can plug in.

It promises one ability: search. Give it words, get back matches.

That is the whole contract — WHAT, not HOW.

 

Now a totally fake librarian named FakeIndex.

We implement the same socket for it.

Its search ignores the real world entirely...

...and just returns one canned hit: the file, a fake snippet echoing your query, and a 0.9 score.

Eight lines. grep calls the shape and never notices it is a puppet — which is exactly the point.

🧪
Why a fake exists

The fake lets the test suite check search logic fast and offline — no real cloud, no network. Any plug that fits the socket works.

The payoff: swap the plant in one spot

One tiny function decides which backend grep uses. Adding offline search later means changing only this — every caller stays untouched.

CODE 
/// Resolve the search backend. Phase 1: always the cloud client. Phase 5 adds
/// local/offline selection here.
fn resolve_index(api_url: &str, key: &str, tag: &str) -> Arc<dyn SemanticIndex> {
    let api = Arc::new(semfs_core::api::ApiClient::new(api_url, key, tag));
    Arc::new(CloudIndex::new(api))
}
PLAIN ENGLISH

A function whose job is just "pick the search backend."

 

It returns Arc<dyn SemanticIndex> — a shared handle to some thing that fulfills the search contract.

Today it builds a cloud client...

...and hands back the cloud backend. Tomorrow, swap in a local one right here — and nothing else changes.

04

The Outside World: Backends and the Couriers That Reach Them

Back in module 1, your write was "durable the moment the prompt returned" — but it had not reached the cloud yet. So who actually drives it there? And what happens if the cloud is slow, offline, or a teammate deleted a file on another machine? Meet the courier fleet.

A shipping logistics network

Your cache 🗄️ is the warehouse next door: instant. The backend 📚 is a distant fulfillment center. Between them runs a fleet of couriers — the SyncEngine 🚚 — each with its own route and schedule, so YOU never wait at the loading dock.

semfs offers three backend tiers. The filesystem behaves identically across all of them — graceful degradation, not lowest common denominator.

📦

Embedded (the default)

SQLite + sqlite-vec + FTS5. Local, offline, one single binary.

Best for: a personal mount on one machine.

🏢

Server

Postgres + pgvector. Concurrent multi-writer, large corpora (HNSW index).

Best for: a team, many documents.

☁️

Cloud

Supermemory API. Zero local index — the server keeps a memory graph for you.

Best for: shared, managed memory you do not host yourself.

The dispatcher has four dials

The SyncEngine reads its schedule from one small settings struct. Two of the dials are timers; two are simple on/off switches that the mount flags flip.

CODE 
/// Knobs for the sync engine. All optional — defaults are production-sane.
#[derive(Debug, Clone, Copy)]
pub struct SyncOptions {
    pub delta_interval: Duration,
    pub deletion_scan_interval: Duration,
    pub pull_enabled: bool,
    pub push_enabled: bool,
}

impl Default for SyncOptions {
    fn default() -> Self {
        Self {
            delta_interval: Duration::from_secs(30),
            deletion_scan_interval: Duration::from_secs(300),
            pull_enabled: true,
            push_enabled: true,
        }
    }
}
PLAIN ENGLISH

These are the dials. Every one is optional — the defaults are sensible for real use.

There are exactly four of them.

delta_interval: how often to check the backend for newer documents.

deletion_scan_interval: how often to check what got deleted remotely.

pull_enabled: master switch for bringing changes IN. --no-sync flips this off.

push_enabled: master switch for sending changes OUT. --no-push flips this off.

The defaults: pull every 30 seconds...

...scan for deletions every 300 seconds (5 minutes)...

...and both switches start ON, so a fresh mount syncs both ways.

🎛️
A mount flag is not magic

A flag just decides which couriers clock in. --no-push means your writes never leave your machine — exactly how you read a shared container without contaminating it.

The five couriers and their schedules

With both switches on, five background loops run on their own timers. Each has one job.

🚚
Loop A — delta pull (every ~30s)

Fetches documents sorted newest-first and reconciles anything newer than our watermark into the cache.

🗑️
Loop C — deletion scan (every ~5min)

Compares all remote IDs against local records and unlinks anything that vanished from the backend.

📤
Loop D — push worker

Sends queued local writes up to the backend. Coalesces rapid edits to at most 2 requests per file. (Deep dive in module 5.)

⏳
Loop E — inflight poller

Watches documents the server is still processing (extract → chunk → embed → index → done) and updates their status by polling.

💧
Loop F — hydration worker

Pulls the actual file contents on demand, only when you open something.

💡
Durable now, delivered later

semfs splits durable (instant, local) from delivered (eventual, remote). Engineers call this write-behind caching — speed and safety without making you wait on the network.

05

The Clever Tricks

You can build something that syncs files. Making it survive flaky networks, rapid-fire saves, server hiccups, and a laptop that sleeps mid-write — that is engineering. Here are the tricks semfs uses.

The rookie courier vs. the veteran

🚲
The rookie

Delivers every package the instant it lands, hits traffic, crashes, loses packages, and re-rides the same street ten times.

🛵
The veteran

Batches deliveries, waits a little longer each time a buzzer is broken, staggers departures, keeps a waterproof manifest, and knows a secret shortcut around the toll bridge.

🚚
Meet the cast (again)

The SyncEngine 🚚 is our courier. The Mount 🚪 is the folder it serves. The Backend 📚 is the warehouse it delivers to. These seven tricks are the craft inside the courier's daily routine.

The seven tricks

📦

Write coalescing

Rapid edits to one file collapse to at-most-2 server requests, no matter how fast you hit save.

⏳

Exponential backoff

When a send fails, wait longer before each retry instead of machine-gunning a struggling server.

🎲

Adaptive interval + jitter

Poll faster after activity, stretch out when idle, and add randomness so mounts do not all call home at once.

🛟

Crash-safe queue

The push queue lives in a database with a journal, so a crash or sleeping laptop never loses queued writes.

🪄

The macOS NFS trick

Apple removed kernel file drivers, so semfs runs a tiny network file server on your own laptop and mounts that.

📝

Agent hint injection

On mount, semfs writes a note into your AI's instruction file telling it to use semfs grep inside that folder.

🔎

Hybrid semantic grep

Search fuses meaning with exact words, so it finds the right line whether you recall the gist or the precise term.

💡
Why you should care

These are exactly the things an AI assistant will not add unless you ask. Knowing they exist turns you from someone who asks for features into someone who asks for robust features.

Trick 2 — Wait longer each time the buzzer is broken

When a building's buzzer is broken, a veteran does not buzz it a thousand times a second — they wait a bit, then a bit longer, giving the resident time to answer. That is exponential backoff.

CODE 
/// Exponential backoff in milliseconds for the Nth failed attempt
/// (attempt=0 → first retry, already-failed once).
fn backoff_ms(attempt: i64) -> i64 {
    match attempt {
        0 => 500,
        1 => 1_000,
        2 => 2_000,
        3 => 5_000,
        4 => 15_000,
        5 => 30_000,
        _ => 60_000,
    }
}
PLAIN ENGLISH

A note: this decides how long to wait before the Nth retry.

"Given how many times we have failed, here is the wait..."

A lookup table from failure-count to milliseconds.

First failure? Wait half a second (500 ms).

Then 1 second, then 2, then 5, then 15, then 30...

...and anything beyond that is capped at 60 seconds.

This gives a struggling Backend 📚 room to recover.

Trick 5 — The secret shortcut around the toll bridge

Apple stopped letting third-party drivers run deep inside macOS. So semfs pulls a magic trick: it runs a tiny network-file-server inside itself, on your own machine, and mounts that.

    /// that replaces FUSE entirely.

PLAIN ENGLISH

This mode uses NFS, which works on both macOS and Linux.

The background program starts a little file server at 127.0.0.1 — your own machine.

Then it asks the NFS client macOS already ships with to mount it.

No kernel extension, no third-party driver to install...

...this is what replaces FUSE entirely on macOS.

Your Mount 🚪 is technically a network drive — pointing at a server inside your own laptop.

🪄
The surprise

Your folder looks completely normal, but under the hood it is a network drive whose "remote" server is sitting on the same machine. No install, no kernel hacks — just a clever use of plumbing every Mac already has.

Trick 3 — Stagger the departures, and an honest confession

If a thousand couriers all leave at exactly 9:00, they jam the same intersection. Jitter staggers them — and prevents the dreaded thundering herd.

🎲
Adaptive cadence

Check sooner right after activity (never under 10s); slow down to 60s after three idle polls — then sprinkle in random seconds.

📝
Agent hint injection — a steer, not a contract

On mount, semfs edits your AI's instruction file to say "search this folder with semfs grep." On unmount, it removes the note.

🤝
Honest engineering

The code admits it plainly: "What this is not: a guarantee... Treat as a steer, not a contract." The AI may or may not obey — and good code is upfront about its own limits.

The caretaker's footprint: four little files

One mount = one tag = one daemon (🧑‍🔧). That daemon drops exactly four files on disk. Learn these and you can debug almost anything.

<cache_dir>/ Everything one mount needs lives under one folder
<tag>.db 🗄️ The Cache — a SQLite database holding your bytes + the write queue
sockets/<tag>.sock The Unix socket — the phone line the CLI uses to talk to the daemon
pids/<tag>.pid The daemon's process ID — its “on duty” badge number
logs/<tag>.log The logbook — your first stop when something is wrong
semfs status Reads the pid file and tells you if the caretaker (🧑‍🔧) is alive
semfs logs Prints the logbook for that tag — what the daemon has been doing

“Is the caretaker alive?” in real code

Here is the exact function semfs uses to check whether the daemon is still running. The trick is delightfully simple — and a little scary-sounding.

CODE 
/// Is the given pid alive? POSIX: shells out to `kill -0 <pid>`, which
/// does not actually send a signal — it's just a liveness probe that
/// succeeds iff the process exists and we're permitted to signal it.
pub fn pid_alive(pid: u32) -> bool {
    std::process::Command::new("kill")
        .arg("-0")
        .arg(pid.to_string())
        .stdout(std::process::Stdio::null())
        .stderr(std::process::Stdio::null())
        .status()
        .map(|s| s.success())
        .unwrap_or(false)
}
PLAIN ENGLISH

The comment is the key: this is just a “is it alive?” probe.

Despite the name, kill -0 does NOT kill anything.

It just asks the OS “does a process with this badge number exist?”

Define a function that takes a signal-free check.

Run the real kill program as a tiny helper…

…with the -0 flag (the “just check” mode)…

…pointed at the daemon's badge number.

Throw away whatever it prints — we only want the verdict.

Same for error text — silence it.

Actually run it and grab the result.

Did it succeed? Then yes, the daemon is alive.

If anything went wrong, assume it is dead (safe default).

Done.

🛡️
Why a subprocess instead of doing it directly?

The whole crate forbids unsafe code, so semfs borrows the operating system's own kill tool rather than poking the kernel itself. Clean and honest.

🚚
Your data was never at risk

The Cache (🗄️) is a SQLite database with a WAL, so a crash cannot corrupt it. Queued writes simply resume and the SyncEngine (🚚) drives them to the Backend (📚).

The Operator's Debug Routine

1
semfs status — is the daemon alive?

This runs pid_alive for you. Dead caretaker? That explains a hung mount instantly.

2
semfs logs — what does the logbook say?

Read the per-tag log (🗄️). Almost every mystery confesses itself here.

3
Read the INFO / WARN / STOP tiers — patient or stuck?

When the Backend (📚) is slowly processing a file, semfs escalates its messages. INFO = normal. WARN = taking a while. STOP = genuinely stuck. This is how you know whether to wait or worry.

4
Stale files after a crash? Cleanup runs on next mount.

cleanup_stale clears the leftover sign automatically — usually nothing for you to do.

5
Still stuck? semfs unmount --force

A force unmount tears down a wedged mount so you can start fresh.

6
Re-mount with the firehose on

Set the environment variable RUST_LOG to crank up log detail: RUST_LOG=semfs=debug,semfs_core=trace semfs mount …
