The workspace map — what actually fits in the agent's context

Not "the knowledge graph" (it's 6–10× too big) and not "just a file list." A layered map: a filesystem skeleton + a distilled community overlay, always cached; the full KG stays on disk as an on-demand lookup. All numbers are measured from the real chanpin-gemma-q4 seed (698 files, 9,300 entities).
The map in one idea: give the agent a cheap, cached table of contents of the workspace so it navigates to the right place itself — instead of relying on the opaque ranked retrieval that buried the right file.

1 · What fits in a 5,000-token budget

Takeaway: the full KG can't go in context — 9,300 entity names alone are ~28k tokens, the edge list ~50k. Only the FS skeleton (1.4k) + the community overlay (2.1k) fit, together ≈ 3.5k.

2 · What is a "community overlay"? (the worked example)

The KG clusters files into communities (Leiden clustering on the file↔entity graph): files that share many entities land in the same community ≈ a topical group. The overlay is just one line per community — a label + size + where it lives. It's the KG distilled to a table of contents, not the raw graph.

# RAW KG — every file and the entities it mentions (9,300 entities total — too big to show/ship) compliance_policy.md → {content governance, grading, China-US, risk} content_grading.xlsx → {grading, classification, risk, audit} risk_audit_2025.pdf → {risk, audit, governance} evaluation_1.txt → {product eval, score, UX} evaluation_10.txt → {product eval, score, rating}
Leiden clustering — group files that share entities
Community C4 = { compliance_policy.md, content_grading.xlsx, risk_audit_2025.pdf } share {governance, risk, grading} Community C0 = { evaluation_1.txt, evaluation_10.txt, … 60 more } share {product eval, score}
label each community (by dominant dir / top entities) → keep only the label
# THE COMMUNITY OVERLAY — what the agent actually sees (1 line each, ~173 lines, 2.1k tokens) C4 (19 files) ~ /chanpin/compliance_and_risk_control/ · governance, grading, risk C0 (62 files) ~ /Desktop/Product_Evaluation/ · product eval, scores C1 (61 files) ~ /chanpin/platform_and_open_capabilities/ · reporting API, data fields C3 (28 files) ~ /chanpin/quality_assurance_and_testing/ · API test coverage …
Why "overlay": it sits on top of the FS skeleton. The FS skeleton says where files are (dirs, counts, types). The community overlay says which files form a topic — and topics can span directories. Together: structure + semantics.

4 · The punchline — the KG is 82% precise pins, 0.3% decoy-magnets

For each entity, how many files does it link to? (Fewer files = more distinctive = a better landmark.)

The 32 generic entities (11+ files) are exactly what the PPR diffusion spread mass through to bury the right file. The prior mixed all 9,300 into one diffused score → the 32 hyper-connected ones dominated → the named file sank to #13.
The map uses the KG the opposite way. The 7,644 single-file entities become precise entity → file lookups (used on demand); the 32 generic ones pin nothing. Same graph — flipped from harmful ranking prior to helpful navigation index.

5 · The layered architecture

◆ In context (cached, ~3.5k tok)

  • FS skeleton — dirs, counts, types, landmark files → where things are
  • Community overlay — ~173 cluster labels → what each region is about
prefix-cached → ≈free after turn 1 · always present, no tool call needed

▸ On disk (queried on demand)

  • Distinctive entity → file lookup (7,644 single-file pins, ~50k tok — never dumped)
  • Content FTS / scoped grep — when the agent drills into a region
the agent calls these after the map tells it where to look
agent reads map (FS + communities) ──▶ picks a dir / cluster ──▶ drills: scoped grep in that dir · entity lookup "where is X" · cat a small file ⇒ decoys in OTHER dirs fall out of scope · the buried file is reachable directly · fewer blind turns
Why this beats the prior: the prior mixed distinctive + generic into one diffused score (generic won → burial). The map separates them — communities for coarse semantic navigation, single-file entities for precise on-demand pins — and lets the agent navigate (its strength) instead of trusting an opaque ranking (the thing that failed).

6 · The realistic middle ground — the enriched overlay

Plain community labels route the agent to a region but say nothing about content (the abc.txt = Anthropic call problem). The fix: per community, append its top distinctive entities (the single-file / low-degree ones — the precise pins, not the generic decoy-magnets). One line now carries structure + topic + content signal.

# PLAIN overlay — region only C1 (61f) /platform_and_open_capabilities/ │ + append each community's top DISTINCTIVE entities (degree ≤ 4) ▼ # ENRICHED overlay — region + content signal (REAL chanpin) C1 (61f) /platform_and_open_capabilities/ · 王芳, Redis会话缓存, RSA非对称加密机制, iOS SDK C4 (19f) /compliance_and_risk_control/ · SGS工作编号, UR研究院, 上海市宝山区漠河路151号 C7 (13f) /desktop/product/ · Product Director, Q21, gmv # illustrative — your example: C12 (8f) /notes/ · sales calls · Anthropic, OpenAI, pricing-tier, Q3-deal ← now "Anthropic" is visible in the map
Budget: the enriched overlay is ~4.2k tokens and carries the directory in each line → it can be the whole map (no separate FS skeleton needed), fitting 5k. Adding a separate FS skeleton pushes it to ~5.6k → then trim to 3 entities/community or merge tiny clusters.
Selection quality is the lever. Notice C0/C2 surfaced bare dates (2024-07-09) — low-value labels. The overlay is only as good as the entity ranking: filter out dates/standalone numbers, prefer named entities (people, orgs, products, ids, tech terms). Garbage entities → a map that misroutes.

The pattern: the enriched overlay solves the navigation failures (burial, under-exploration, affordance, "what to grep") cleanly, and partially the content-precision ones (opaque names, generics, duplicates) by routing the agent to the right region where a cheap scoped grep finishes the job. It does not touch the harness output bug or replace per-file summaries — those are separate levers.
