Two honest cases: semfs single-vector+KG  vs  next-plaid ColBERT multi-vector

Each side argued in its strongest form (Rapoport's rules: state it so well the advocate says "I wish I'd put it that way"), then rebutted, then synthesized — all grounded in the WB-Lite GLM runs.

The one-sentence tradeoff

One vector + structure bolted on the side (semfs: a single dense vector per chunk, with a hidden KG graph and FTS5 doing the structural work at re-rank time)  vs  many vectors + structure baked into the embedding (next-plaid/ColBERT: ~300 token vectors per doc, late-interaction MaxSim, and filename/metadata/code-structure embedded up front).

advocate voice

The strongest case FOR semfs's design

"Retrieval is not the bottleneck — and a single bounded vector plus a graph is the *right* amount of machinery for an agent filesystem. Our own data proves it: when we cleaned the judge, the hidden-KG prior beat plain by ~7pp (26% vs 19%) on 3 of 4 personas, and it did so while using *fewer* tokens, with vec0 linked straight into the binary — nothing to ship, nothing to serve. The fancy graph variant (PPR diffusion) bought *zero* accuracy over the trivial 1-hop prior. That's the whole thesis: marginal retrieval sophistication has near-zero marginal return here, because an agent re-queries, reads, and reasons — it doesn't need a perfect rank-1, just the right file *somewhere* it'll open. Single-vector keeps indexing cheap, updates incremental, and the system explainable. Every dollar of complexity we *didn't* spend on multi-vector, we can spend on the things that actually moved the numbers: delivery format, judging, and agent behavior."

Core insights it gets right

  • Simplicity that survives. One vector per chunk = trivial to index, update, debug, and ship (vec0 in-binary, single SQLite file).
  • The agent is forgiving. It re-greps and reads; "good file in the pool" beats "perfect rank-1." Late-interaction precision is partly wasted on a re-querying agent.
  • Structure as a *cheap prior*, not a cost center. The KG graph adds the relational signal at re-rank for a couple of SQL joins.
  • Right place to spend. Our bottlenecks were downstream (judging/delivery/behavior); a leaner retriever frees budget for them.

Evidence from our runs

KG prior beat plain +7pp (26.1% vs 18.8%) on 3/4 personas, at ≤ plain tokens.
PPR diffusion gave 0 gain over 1-hop → sophistication had no payoff.
The biggest accuracy swings came from judge/delivery bugs (filename, dir-only: ~2× swings), not retrieval quality.
advocate voice

The strongest case FOR next-plaid / ColBERT

"Your single content-only vector is structurally blind, and you have the receipt: kaifa — your only *code* corpus — is the one persona where semfs *loses to plain*, because the answer sat in a folder named Not_Debugged/ and your embedding never sees a filename. That isn't a tuning gap; it's an architectural ceiling. ColBERT fixes it at the root: it embeds the filename and metadata, and ColGREP (built on exactly this engine) extracts signatures, docstrings, and call graphs *before* embedding — it is purpose-built for the corpus you fail on. Late interaction also recovers what a single 768-d vector throws away: a 300-token document keeps 300 vectors, so a rare query term still finds its needle instead of being averaged into mush. That's why ColBERT tops single-vector on hard benchmarks. And it's still local-first — IVF + 2/4-bit PQ + mmap put a million docs on one machine. You're hand-building, at re-rank time (KG graph, FTS, RRF, PPR knobs), a worse approximation of the structural signal ColBERT captures for free at embed time."

Core insights it gets right

  • Embed the structure, don't bolt it on. Filename + metadata + code signals go *into* the vectors — exactly the kaifa blind spot.
  • Single-vector loses information. Averaging a doc into one vector buries rare-but-decisive terms; MaxSim keeps them.
  • Purpose-built for code. ColGREP = semantic code search; kaifa is a code corpus.
  • Still local-first. PQ + mmap → millions of docs on CPU; not a cloud dependency.
  • Metadata prefilter. SQLite WHERE/REGEXP narrows before scoring — structure as a first-class filter.

⚔️ Now address each strong version

Rebutting the steel-manned semfs case

"Retrieval isn't the bottleneck" — true for the *document* personas, but kaifa is a clean counterexample where it is, and the architecture (not a bug) caused it. "Spend elsewhere" doesn't fix a structural ceiling.
"The agent is forgiving" — forgiveness has a price: on kaifa it re-grepped 563× and burned +237k tokens precisely because rank-1 was wrong. A better rank-1 would have *saved* the tokens the simplicity argument claims to protect.
"PPR gave no gain, so sophistication doesn't pay" — that indicts *graph diffusion*, not *better embeddings*. PPR re-ranks the same pool; ColBERT changes what's *in* the pool and how it's matched. Different lever.

Rebutting the steel-manned next-plaid case

"Late interaction lifts accuracy" — maybe, but our measured ceiling was set by delivery & judging bugs (filename, dir-only, nondeterminism: ~2× swings). A better retriever can't lift an accuracy number that a broken judge is capping.
"Embed filename + metadata" — the *insight* is right and we should steal it; but you don't need 300× the vectors to get it. Prepending the path to the chunk text captures most of the kaifa win at ~zero migration cost.
"Still local-first" — true, but ~300 vectors/doc + IVF/PQ codebooks + mmap is materially more operational surface than one SQLite file with vec0 in the binary. For an embeddable agent FS, that footprint is a real cost.
"Single vector loses rare terms" — partly mitigated already: semfs runs hybrid (dense + FTS5 keyword via RRF), so exact rare terms are caught by the lexical lane. The gap is narrower than pure dense-vs-ColBERT.
what I actually believe after both

Synthesis — take the insight, not (yet) the engine

Steel-manning changed the question. It's not "single-vector vs ColBERT." next-plaid's winning argument is narrow and specific: embed the structure (filename + metadata). Its weaker argument is the expensive part: switch to 300-vector late interaction. semfs's winning argument is don't pay for retrieval sophistication that doesn't move the measured number. Both are right about different things.

The staged plan (cheap → expensive, each gated)

  1. Now (hours): prepend file path/name (+ extracted summary/metadata) to each chunk before embedding — sqlite_vec.rs:591. Re-run kaifa. Prediction: the plain-beats-semfs gap closes. This tests next-plaid's *winning* claim without its *expensive* one.
  2. Then: fix the delivery/judging ceiling (recurse-judge done; cache_read accounting; n≥2 judge) so accuracy numbers are trustworthy enough to compare retrievers at all.
  3. Only if (1)+(2) show retrieval is still the residual gap: pilot next-plaid/ColBERT on the *code* lane (kaifa) as an isolated A/B — the corpus it's built for — with a clear rollback. Don't rewrite the whole store to find out.
