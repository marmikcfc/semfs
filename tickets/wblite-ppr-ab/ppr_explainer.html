How semfs ranks — and what ppr_off · ppr_on · ppr_map actually do

An interactive walkthrough of the query flow and the three graph-prior settings. Start simple (ELI5), climb to expert. Every number is from the WB-Lite GLM runs.

① The idea, in 5 zoom levels

② The query flow (the part that's the SAME for all three)

Press Run query and watch a question travel through semfs. The three settings only change ONE box — the graph prior.

idle

Key fact: the prior is bounded (it can add at most ~0.15 to a file's score) and it only re-orders files already found by search. It never adds a new file. So it can only help by pushing an already-retrieved good file higher.

Query: "Shengye Electric employees education qualifications" — the answer lives in shengye_electric_2024_annual_report.docx.

ppr_offannual_report (direct entity), hr_records (1-hop)✓ answer is direct — boosted to top2 SQL joins
ppr_onsame + org_chart, training_log, payroll (2–3 hops, faded)✓ same answer — extra files add noise, not signal30 PageRank iterations / query

The answer was 1 hop away, so PageRank's multi-hop reach found nothing new — it only diffused mass to unrelated files. This is why the theory ("multi-hop is richer") doesn't win in practice on these corpora.

⑤ Why ppr_map loses despite using fewer turns

The map does help navigation (−20% turns). It loses for one reason: the model has no memory of it (cache_read=0), so the 5,415-token map is re-sent and re-billed on every single turn. Press play:

Same result, less machinery. ppr_on and ppr_off land on the same answers (the prior is bounded + only re-orders the pool), so accuracy (26.1 vs 26.0) and tokens tie. But ppr_on pays for it: +13% latency, three tuning knobs, and it slightly dilutes on big corpora (ppr_off ≥ ppr_on on 3 of 4 personas; per-case off 28 / tie 27 / on 27). PageRank's multi-hop advantage never materializes because answers sit within 1 hop. → Choose ppr_off. Drop the map until cache_read is surfaced.
Not universal: houqin prefers ppr_on (+4.3) and even plain; kaifa (code) costs more tokens than plain. GLM, n=2–3, noisy judge — a direction, not a final verdict.
