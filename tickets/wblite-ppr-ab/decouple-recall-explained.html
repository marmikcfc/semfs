Decoupling recall from the prior

Why a file that literally contains your answer can get thrown away — and the deterministic fix. Grounded in the WB-Lite PPR run (grep.rs, cases 358 & 139).
Recall = "did the right file get returned at all?"  ·  The prior = the knowledge-graph's guess about which files matter. Today one number does both jobs — order and keep/drop — so the graph's guess can delete the file you need. Decoupling = the prior may only reorder, never delete; membership is decided by a separate deterministic "does it literally match?" rule.

The overconfident librarian

Explain like I'm 5

You ask a librarian for "the book about dolphins." She does two things: she picks which books to put on the table, and she arranges them in an order.

The problem: she's a know-it-all. She also hides books she personally thinks you won't like — and one of the hidden ones is the actual dolphin book. You never even see it, so you go home with the wrong book.

The fix: make a rule — every book whose title says "dolphin" MUST go on the table. The librarian can still arrange them in whatever order she likes (she's smart about order!), but she is not allowed to hide any book that matches your word.

Two jobs got tangled: "which to show" and "what order." Untangle them: matching decides which; the smart guess only decides order.

Finding vs ordering are different questions

Explain to a high-schooler

Two separate questions when you search:

① Recall — did we FIND it?

Of all the files that actually contain the answer, did the right one end up in the returned list at all? This is about membership.

② Ranking — what ORDER?

Given the returned list, which goes first? This is about arrangement. Getting order wrong is cheap (scroll down a bit); getting recall wrong is fatal (it's just gone).

The prior is a guess from the knowledge graph: "files connected to lots of related stuff are probably relevant." Useful for order. But right now that guess is mixed into the one score that also cuts the list short — so a low guess drops the file. A file you could have read becomes a file you never saw.

Decouple: let a dumb-but-reliable check ("does the file literally contain the words you searched?") guarantee the file is kept. Let the smart guess only decide where it sits in the order.

One score is doing two jobs

Explain to an engineer

Here is the actual semfs grep pipeline. The bug is that the KG prior is fused into the single similarity that drives both ranking and the cutoff:

query │ ├─ lexical lane (substring / FTS — deterministic, 100% recall) ├─ dense lane (embeddings) └─ KG-prior lane (PPR / 1-hop graph mass) │ ▼ RRF fuse → ONE number: hit.similarity │ ▼ rank by similarity grep.rs:1115 adaptive_k(similarity) │ ▼ take(render_n) ← TRUNCATE grep.rs:1191 hits.take(render_n) │ ▼ render to agent BUG: the prior is inside similarity, and similarity drives the cut. → a strong lexical hit with a weak prior → low fused score → ranked low → dropped by take(). The agent can't read what it never received. Recall is now hostage to the prior.

Decoupled design: tag which hits are literal matches at fuse time; exempt them from take(). The prior still reorders, but the cut only trims the fuzzy (dense/KG-only) tail.

returned = ALL literal hits ∪ top fuzzy hits (filling remaining budget) └ prior may reorder this set, but may NOT remove a literal hit guarantee: literal-match(file, query) ⇒ file ∈ returned (deterministic)

Don't let the re-ranker prune the candidate set

Explain to a search/IR designer

Standard two-stage retrieval has a strict division of labor:

Stage 1 — candidate generation

Optimized for recall. Cheap, deterministic, recall-complete (BM25 / substring / FTS). Job: never miss a plausibly-relevant doc.

Stage 2 — re-ranking

Optimized for precision. Expensive, smart (dense, cross-encoder, KG prior). Job: order the candidates well.

The cardinal rule: Stage 2 reorders Stage 1's set; it must not shrink recall below it. Our bug is that the KG prior (a Stage-2 signal) is folded into the fused score that performs the presentation cutoff — so the re-ranker is silently pruning the candidate set. Any cutoff K (6, 10, 20, ∞) is then a heuristic window on a list the prior already corrupted: widening K only widens a blind spot it can't remove.

Decoupling restores the contract: recall is owned by the deterministic lexical lane and is a lower bound the re-ranker cannot violate. Formally — let L(q) = files with an exact lexical/name match. Enforce L(q) ⊆ returned(q) always; the prior chooses the permutation of returned(q), never its membership. Cutoff applies only to returned \ L(q).

case 358 the named CSV sat at rank #12–15 (prior promoted thematic *summary*.md neighbors). It is a literal match for the task → under decoupling it is in L(q) → returned with probability 1, no matter its prior rank.

Never put a corruptible prior upstream of an irreversible cut

Expert · failure-mode & theory

Frame the prior as a Bayesian prior over relevance and the fused score as a posterior. Multiplying the prior into the retrieval score makes it a gatekeeper: when the prior is mis-specified, it drives the posterior of a genuinely relevant doc toward zero. On dense / decoy-laden graphs the prior is mis-specified — co-mention/community mass concentrates on thematic neighbors, not on the literally-named target — so coupling propagates the prior's mis-specification directly into recall loss.

Recall loss is special because truncation is a lossy, irreversible channel. A ranking error is recoverable downstream — the agent can scan three more lines. A dropped document is unrecoverable — there is no operation the agent can perform on a result set to retrieve a file that isn't in it. The design principle: never place a corruptible, mis-specifiable signal upstream of an irreversible operation. Keep the prior on the reversible side (order); keep the irreversible side (membership/cut) on a signal you trust.

Why "literal match" is the right trusted signal: presence of the query terms is a near-sufficient statistic for "this file could contain the answer," and it is a deterministic function of (query, corpus) — identical inputs → identical recall set, no embedder drift, no graph density effects, no adaptive-confidence variance. That determinism is exactly what the question "how do we deterministically solve it?" demands: P(answer_file ∈ returned │ answer_file literally matches q) = 1.

The one residual the literal floor can't fix

Near-duplicate decoys (case 139: 4+ content-identical event_plan_17.md in parallel dirs + a trap dir). When candidates are content-identical, the literal signal returns all of them and cannot rank them apart — no content signal can, by definition. The disambiguator is structure, not content: enumerate one canonical directory (ls dir && read dir/*) — which is exactly how the plain agent wins. So the complete fix is two-pronged: literal floor for thematic burial + directory enumeration for duplicate decoys.

Net: "decouple recall from the prior" = move membership decisions off the mis-specifiable, fused, truncating score and onto a deterministic literal lane that the prior may permute but never prune. The prior becomes a tie-breaker; recall becomes a guarantee.
