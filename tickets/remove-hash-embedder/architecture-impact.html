Remove HashEmbedder — architectural impact

What changes when the fake embedder is dissolved. The thesis: one knob that pretends to be a model selector is secretly the search router. This splits it into two honest, orthogonal axes. Grounded in the verified code on feat/backend-agnostic-store — every file:line below was checked against the tree.

Headline: this is a control-plane refactor, not a data-plane one. The storage schema has zero coupling to hash — no re-index, no migration, no stored-vector change. Only config / routing / wiring moves.

1 · One knob becomes two

Today a single setting pretends to choose a model but actually decides where search runs:

Today — "pick the hash model" → really means → "turn off local search, use the cloud"
You say "give me garbage vectors" when you mean "search the cloud." The model axis is secretly the router.
▼ split into two honest questions
After — Axis A · SEMFS_EMBED_BACKEND = how vectors are made
local | openai | openrouter — real embedders only. hash is gone.
After — Axis B · SEMFS_STORAGE_BACKEND = where search happens
sqlite | pgvector | pglite | cloud — cloud becomes a first-class storage choice. Routing keys off this, never the embedder.
Net: we don't add a component — we dissolve one. Fewer concepts, each with a single responsibility.

▸ resolve.rs:233 — the "pgvector + hash = misconfig" guard

Why it's a smell: the embedder axis is steering the storage axis, and cloud isn't even a StorageChoice — it's an emergent side-effect of choosing a fake embedder.

4 · The dependency flip (the real structural win — DIP)

Today the caller must resolve an Arc<dyn Embedder> for every path — so the cloud path is handed a fake HashEmbedder it never uses. After: the embedder retreats into the local store constructors; the mount depends only on the abstraction it actually uses.
