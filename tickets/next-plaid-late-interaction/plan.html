④ The two flows — seed creation (embedding) + query

Does next-plaid need an initial embedding? — Yes, always. The flow you shared is it: corpus → tree-sitter → structured text → [model] multi-vector embedding → NextPlaid index is the build/embedding stage; Search only runs once that index exists. "Incremental" = you can add files later without a full re-embed — the initial corpus must still be embedded first. For us that's the seed-creation stage, done off-box and baked into the E2B template.

① Seed-creation / embedding stage (offline · one-time · baked into the E2B template)

Config A — unified: 3 routes → ONE model → ONE index (no second model, no merge)

This is the diagram you circled. Three input types fan out to three parsers, then converge into ONE LFM2 encoder and ONE shared index. This is one of three options — Configs B & C (below) split into a code lane + doc lane.

Config A in one line: one LFM2 encoder, one shared centroid space → code-unit and doc-unit scores are directly comparable under one MaxSim, no merge. Caveat at k-means: centroids train over all tokens, so a code minority can quantize worse. This is the config that did NOT move kaifa under PPR-style ranking — so for kaifa we look at B / C, not A.

Config B / C — split lanes: a code lane + a doc lane → 2 indices → merge at query (this is what you asked to see)

Two independent build lanes, two indices. At query time the question is encoded and searched in both, then fused with RRF. B = same model (LFM2) in both lanes. C = a code model in the code lane + LFM2 in the doc lane (= 2 models).

② Query stage (online · in the E2B cell, per search — no re-embedding of the corpus)

The embedding flow up close — and where LFM2 plugs in

The encoder (per-token, dim 128)

tokenize → insert marker token after CLS ([D] for docs, [Q] for queries) → ONNX/PyLate forward → projection + per-token L2-norm inside the graph → output [n_tokens, 128].

  • Doc: skiplist removes punctuation tokens → variable length (≤ document_length 512).
  • Query: MASK-expand to fixed query_length 32, attn=1 (so short queries still get rich matching).
vs ours today: gemma emits one pooled 768-d vector/chunk; here it's ~N×128 per token.
LFM2 has no ONNX — two ways to feed it
  • (a) ONNX-export via pylate-onnx-export → model.onnx, so colgrep encodes internally. ⚠️ hybrid backbone = export risk.
  • (b) PyLate-direct (PyTorch) → precomputed vectors → next-plaid update API (base64 LE-f32). No ONNX.
The encoder slots in at BUILD stage ④ and QUERY stage ②. Everything downstream (centroids/PQ/IVF/MaxSim) is identical either way.

Verified from the clone: streaming build pipeline colgrep/src/index/mod.rs:494–715 (tokenize → encode → pool → PQ → write); encoder next-plaid-onnx/src/lib.rs:1118 encode_documents / encode_queries; index build MmapIndex::create_with_kmeans; precomputed-embeddings ingestion via the API update (base64 LE-f32, SDK client.add routes text→update_with_encoding, vectors→update).

④b Contrast — what an English-only (code + docs) workspace would look like

Our corpus is Chinese + mixed, which forces a single LFM2 index (the English code model can't read Chinese comments). If the workspace were English-only with both code and docs, the language handicap disappears and the architecture flips: multi-model becomes the clean default, not a gated escalation.

Why it flips: the only reason we don't reach for LateOn-Code today is its English training on a Chinese codebase. Remove the language mismatch and the code lane is strong, the doc lane is strong, and the only cost left is the RRF merge — which is worth paying when both legs are at full strength. English-only is the world where "use multiple models" is clearly right.

⑤a What changed once Configs B / C existed?

Original view
  • Implementation Option A looked almost identical to "just run colgrep" over one index.
  • Retrieval Config A was the clean first read: one model, one centroid space, no merge logic.
  • Option B was clearly premature before any benchmark result.
New view after adding split-lane configs
  • Implementation Option A still stays for the test, but it is no longer "pure colgrep" for every cell.
  • Configs B/C force extra A-side logic: route corpus into code/doc dirs, build two indices, and add a query-time RRF merge shim.
  • Option B becomes more justified afterward, because the merge logic and split-lane idea now have a direct production home inside semfs grep.

So the recommendation shifts in nuance, not direction: Implementation Option A is still the right benchmark path because it keeps the science cheap and keeps us on their real engine. But Configs B/C make the eventual production path much more likely to be Implementation Option B, not because B helps the test, but because the test now creates reusable merge/index-routing logic that ports naturally into semfs.

⑤c Execution roadmap — now → results on all 3

Decision: ONNX for both (LFM2 exported via pylate-onnx-export; LateOn/LateOn-Code shipped ONNX; onnxruntime everywhere — no PyTorch in the cell). Readiness: the plan is complete; not yet done = the 3 Phase-0 checks + rrf_merge.py. The 3 builds parallelize but share a Phase 0, so: do Phase 0 once → fan out. Wall-clock ≈ 3.5–5 days.

Build scripts? — mostly no. Index build = colgrep init per lane (once ONNX + corpus dirs exist). The one custom script is rrf_merge.py (Config C) — and it's test-time (Phase 2), not build-time, so it doesn't gate building. Phase-0 verifies "colgrep loads our custom LFM2 ONNX"; if it can't, fallback = next-plaid API + a tiny feeder script. Phase 2's arm-wiring is shared (write once); the runs parallelize. Critical path: P0 (LFM2 export) → slowest P1 build → P2 run+debug → P3.
