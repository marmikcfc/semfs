xAFS: what went wrong

We tested whether a semantic file system helps an AI agent answer questions about messy personal file systems. Plain grep won — more accurate and 58% fewer tokens. Here's the test, and the three reasons why.

⚠ Update · 2026-07-01 · this page is superseded at scale

The "plain wins" story below was a small-workspace artifact.

This page tested only vector search (no KG) on a few small workspaces. Once we built the KG / PPR arms and ran the big workspaces, the result inverted — the semantic filesystem wins where it was supposed to: large corpora where brute-force grep drowns.

0 The 30-second version

What we hoped

An agent that searches by meaning (semfs) should beat an agent that crawls files with grep (plain) — especially as folders grow to thousands of files.

What happened

The questions are mostly "find this exact ID / number." Grep is built for exact matches; semantic search finds vibes-similar files and misses the exact one. So plain won.

🔑 One sentence: semantic retrieval shines for conceptual questions; xAFS asks exact-lookup questions — and that's literal grep's home turf.

1 What we actually tested

The benchmark, the two agents being compared, and how a single question is scored.

The benchmark: supermemory/xAFS

13 synthetic people (dp_001 … dp_013), each a realistic personal file system — emails, chats, invoices, medical notes, legal pleadings. Crucially they scale from 5 files to ~10,000 files. Each has questions answerable only by reading the right file (e.g. "What was the Stitch invoice amount?" → gold $2,034). 110 questions total.

Agent gets the raw file tree. Finds answers with real find grep -r cat. This is the "standard filesystem agent."

3 Finding 2 — the combined-seed confound

A methodology bug we caught early: we'd put all 13 people in ONE index.

❌ What we did first

One seed for all 13 workspaces. So when the agent searched for dp_005's Zelle number, semfs grep searched all 19,170 files of all 13 people.

dp_005 (50 files) → 487K tokens
the agent waded through 13 strangers' files

✅ The fix

One index per workspace. The agent for dp_005 mounts only dp_005's seed → searches only those 50 files. (This is how xAFS is designed — each person is isolated.)

tiny per-dp seeds (4–135 MB)
also fixed our disk + sandbox limits as a bonus
⚠️ Without this fix, any "semfs costs more" number was an artifact of searching 13× too much, not the retrieval method. The tight leash caught it before a 660-cell run.

4 Finding 3 — semantic search < literal grep

The real reason it "went wrong." Even fairly scoped, semantic search lost — because the questions are exact lookups.

Climb from ELI5 → expert

🧒 ELI5 📘 Beginner 🛠️ Practitioner 🔬 Engineer 🎓 Expert

Imagine asking a librarian for "the book with the exact phone number 555‑0192." A keyword librarian flips to that exact number fast. A meaning librarian brings you books about phones — close, but not the one with the number. xAFS keeps asking for the exact number.

Plain uses grep -r "Zelle" — it matches the literal string and finds the file containing the confirmation code. semfs uses vector similarity — it ranks files by topic, so "Zelle confirmation number" pulls up Venmo logs and bills (same topic, wrong file).

For an exact-lookup task the answer is a token (an ID, amount, code), not a concept. Embedding similarity blurs tokens together (Zelle ≈ Venmo ≈ payment), so the target file doesn't rank #1. The agent reads the top semantic hits, doesn't find the value, and thrashes — more tool calls, more tokens, often gives up.

Our seed has a vector lane and a BM25/FTS keyword lane. In practice the semfs grep output was vector-dominated — the keyword match for "Zelle" wasn't surfaced above the semantically-near distractors. So the hybrid wasn't behaving like keyword-first for this query class. (Untested: re-weighting FTS could recover it.)

The benchmark's task distribution matters more than the retrieval engine: xAFS is dominated by single_hop exact-extraction. Semantic FS's documented win is largely token efficiency (we saw −30% on several cells), not accuracy on exact lookups. The remaining lever is the KG/PPR prior — entity edges pointing straight at the answer file — which we did not build (needs the GPU). That's the only mechanism that could flip exact-lookup accuracy.

🔫 The smoking gun: dp_005, the Zelle question

Question: "What was the Zelle confirmation number on Yael's transfer?" Gold: ZE9521-09BB

semfs grep "Zelle confirmation number transfer" → returned:

📄 /messages/venmo-log-sept-oct.md ← Venmo, not Zelle
📄 /house/bills/september-bills-summary.md ← bills
📄 /messages/imessage-yael-wren-…md ← chat
…the file with ZE9521-09BB never surfaced.

plain grep -r "Zelle" → found it on the first try.

Result: semfs spent 554K tokens / 58 tool calls and failed; plain spent 75K and got it right.

6 So what now?

✅ Banked

  • Full E2B pipeline validated end-to-end (bake → mount → codex → Gemini judge)
  • Supermemory-faithful judge (run_judge_xafs.py, 9/9 tests + live)
  • Plain baseline = SMFS reproduction; per-dp index fix; semantic<grep finding

🔓 Open (deferred)

  • KG / PPR arms — graph edges to the answer file; the only thing that could beat grep on exact lookups. Needs the GPU gemma-vLLM KG build.
  • FTS/hybrid weighting — make keyword match win for exact lookups before any rebuild.
  • dp_010–013 per-dp seeds (CPU+summary embed too slow).
