Case 289 — Deep Failure Analysis

Workspace-Bench "best-selling product" · codex GPT-5.4 · semfs local vs Supermemory cloud · 2026-06-06

One-paragraph truth. The task asks codex to find a Chinese product-data file using an English query. Local semfs fails to surface it cleanly for three compounding reasons — a cross-lingual recall miss (English query vs 100%-Chinese content), a post-rerank ranking bug that destabilises the answer's position, and a whole-doc return format — so codex compensates with 17–19 tool calls (incl. a 62 KB filesystem walk). Cloud does it in 3–4 calls because it translates the query and returns the answer at a clean #1. Token gap: cloud 18 K vs local best 82 K.

3 · The salience/co-mention bug (F5/F6) — the "why doesn't rerank stick" answer

After L5 the answer is correctly #1 (score −0.719). Then:

// rank.rs — applied AFTER rerank, on the cross-encoder score
h.similarity *= 1.05;                      // L7 co-mention "boost"
h.similarity *= salience(now,last,count);  // L6, salience ∈ [0.85, 1.50]

Cross-encoder scores are negative. Multiplying a negative by a factor >1 makes it more negative → ranks lower. So a "boost" demotes the best hit, while a stale file (salience<1) gets promoted. And because access_count is incremented on every search (F6), the factor drifts run-to-run — the same query yields different top-N orderings. Codex sees an untrustworthy #1 → explores.

Fix: apply L6/L7 as additive nudges, or on a sigmoid-normalised [0,1] score, or before rerank (on positive RRF scores). And don't let read-path access_count bumps feed the same-session ranking.

6 · How to give LOCAL a cloud-style overview (profile.md)

Why it matters. The mount hint tells codex: "read profile.md for an overview." But profile.md is fed by Supermemory's /v4/profile memory API — and it's empty for both cloud and local here (this container has no extracted memories). So that wasn't the cloud/local differentiator — but a populated local profile is still the cleanest way to kill the 62 KB os.walk: give codex the tree up front so it never enumerates it.

Design (proposed, no cloud dependency)

  1. Structural section (deterministic, from the cache): directory tree (depth ≤3) with per-dir file counts + extensions, generated from fs_inode/chunks. This is exactly what the os.walk produces — pre-compute it so codex doesn't.
  2. Content section (cheap LLM, one-time at index): 1-line topic summary per top-level dir from a sample of chunk texts (gpt-4.1-nano) — mirrors Supermemory's "memories" but offline. Links to the summary-augmented-table-retrieval idea.
  3. Wire-up: ProfileFile::warm() falls back to format_local_profile(db) when /v4/profile is empty/unavailable (local mounts). No new flag.

Expected effect: removes F8's 62 KB os.walk and the inspection calls → fewer tool calls → lower tokens, independent of retrieval quality.

7 · Sparse embeddings (replace BM25)?

Probed, not built. BGE-M3 sparse (FlagEmbedding) ranked the answer .txt at #8–11 (pure sparse), dashboard #1 — not a clear win, and it needs a whole new index lane + re-seed. Given today's findings (BM25 already matches the verbatim ZH terms after rewrite; the binding constraints are F5/F6/F8, not lexical recall), a learned-sparse lane is lower priority than fixing the ranking bug and the agent-overview. Revisit only if cross-lingual recall (F1) becomes the bottleneck after F5/F6 are fixed.
