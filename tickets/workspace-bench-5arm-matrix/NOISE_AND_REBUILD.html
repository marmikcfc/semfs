Part 1 · Noise

Where the noise comes from

n=1 is a coin flip

The agent (codex) is a dice-roller, not a calculator. Give it the exact same task twice and it doesn't do the exact same thing — it might search 5 times one run and 30 times the next, find the answer file early once and late the next. So two runs of the same setting can score differently. That gap between two identical runs is the noise. It's not a bug in semfs; it's the agent being a probabilistic model.

Our sharpest example is case 45. On the identical config it scored 3/38 in one batch and 11/38 in another — that's roughly 8% → 29% rubric with nothing changed. Case 53 does the same thing for the plain baseline. When one case can swing ~20 points on its own, a 2–3 point difference between two configs is inside the noise floor — you cannot tell skill from luck without repeats.

Cost is dominated by turn count, not by "search" in the abstract. Codex re-pays a large slice of its context every turn (cache-read), so:

total tokens  ≈  turns × per-turn-context        corr(calls, total) = 0.82

The agent's call count is itself stochastic — we've measured the same config emit 5, 16, and 32 tool calls on different runs. That alone swings tokens by ±30% and accuracy by ±1 rubric point. So the noise lives on both axes at once: a lucky-short run looks cheap and accurate; an unlucky-long run looks expensive and noisy.

Some "noise" was a judge artifact. The big one is SOLVED for every run discussed here — don't double-count it:

  • Filename lottery SOLVED for current runs — ~34/100 WB cases hard-code the expected output filename judge-side only; the agent was never told it, so correct content under the "wrong" name scored 0. Fixed agent-side: run_matrix.py:154 injects the filename into the prompt (WB_OUTPUT_FILES → [REQUIRED OUTPUT FILENAME]) for every cell. Verified: all 10 plain cells and all 10 p2b cells carry expected_files, and the agent followed the hint in 19/20 (followed=True). So both arms got the fix — the p2b-vs-plain comparison is clean on this axis, not pre-fix-vs-post-fix.
  • Trace-too-large (>~320K) silently breaks the LLM judge → we cap inputs.
  • Judge base-URL unset → silent mis-score.

Two residuals remain — but neither is "the lottery." (1) The agent occasionally disobeys the hint it was given — pm_codex_55_nokg_rp2bA has followed=False. That's not a judge artifact; it's the same agent stochasticity from the rungs above (it had the filename and still wrote the wrong name once). The judge-side content-match + n≥2 is the backstop for that. (2) Historical pre-fix cells (the 2026-06-10 5-arm matrix, Modal runs) still carry the raw lottery — only a problem if you mix them into a comparison. For this sweep, treat the lottery as fixed.

Bottom line, stated as the rule from analyze-benchmark-results: n=1 lies. A single per-cell number is a coin flip, not a result. With cases that swing ±20 points alone, the decision rule has to be:

require n ≥ 2 before declaring a win
a config wins a case only if  accuracy ≥ baseline (±1 rubric pt)  AND  tokens lower
lower tokens + lower accuracy = LOSS, not a saving

That's why the headline claim below is hedged: p2b's accuracy edge over plain leans on case-45's coin-flip, so the robust claim is "~3–4× cheaper at not-worse accuracy" — not "more accurate."

Part 2 · Rebuild

Why some knobs need a rebuild and others don't

Every knob in this sweep lives at one of two layers. The layer decides whether I can flip it from Python in seconds — or whether it has to be compiled into the binary that's pre-baked into the E2B template.

Some knobs are stickers — I peel and re-stick them at runtime (an environment variable). Free, instant, no tools. Other knobs are soldered into the circuit board — to change them I have to re-melt the board (recompile the binary). The offload behavior is soldered. That's the whole reason I'm asking for a rebuild: I've run out of stickers and the next idea is under the solder.

I verified this directly: SEMFS_RESULT_LIMIT is read at sqlite_vec.rs:75, the caps at grep.rs:422/470, and SEMFS_REWRITE at grep.rs:910 — all std::env::var. The whole-doc reconstruction at sqlite_vec.rs:62 is plain logic, no env switch.

The env axis is converged. Across the sweep, no env config beats the thorough nokg baseline at held accuracy:

  • h1 tight-caps → starves accuracy (cuts case 53's needed reads). Floor hit.
  • h2 fewer-hits → same starvation, milder. Neutral-to-worse.
  • h3 no-rewrite → neutral (rewrite wasn't the lever).
  • p2b turn-brake-strong → cheapest config that still beats plain, but trades some accuracy for the token cut.

So we have two local optima, both beating plain, and tuning env vars won't move us off them:

thorough  (nokg baseline)   13.4% acc   279K tok    ← max accuracy
lean      (p2b turn-brake)  10.4% acc    90K tok    ← cheapest still-beats-plain

The remaining token sink is whole-document delivery. Today a grep hit reconstructs and returns the whole doc text for the top-N — a single broad grep can return a huge blob, and that blob re-enters context every subsequent turn. The idea (N4 · offload): return a pointer + a tight snippet, and let the agent read the rest on demand only if it needs it. That cuts the per-turn context, which is the thing multiplying by turn-count.

But "return a pointer instead of the whole doc" is exactly the sqlite_vec.rs:62 reconstruction logic — there is no env var that toggles it. To test it I have to change the Rust and recompile.

The reason a code change is heavier than it sounds here is the cross-arch deployment chain. E2B is x86_64; this Mac can't cross-compile fastembed/ONNX. So:

edit sqlite_vec.rs │ ▼ tar src ─▶ Modal x86_64 build (rust:1.95-slim, ~min) ──▶ semfs-fixed binary │ ▼ modal volume get ──▶ benchmarks/e2b/assets/ │ ▼ re-bake into E2B template (boot_prep push) │ ▼ re-run the sweep · n≥2

That's why I asked before doing it: env knobs were free to sweep, but the offload test spends a Modal build + redeploy + a fresh n≥2 matrix. It's the first knob in this campaign that isn't free, so it's a real "do we want to spend this?" decision rather than something I should just burn tokens on unilaterally.

Part 4 · Stateful fix

Why p2b is a patch — and the deterministic fix

p2b tells the agent in words "stop re-searching." But the agent is a dice-roller (Part 1): under a "~5 call" budget it issued 9 discovery calls on case 53 and still scored 0/22. You can't fix a determinism problem with a probability. The real cause is structural — and so is the real fix.

Is the system stateful? Two stores, disconnected.

┌─ AGENT LOOP (codex) ───────────────────────── STATEFUL ─┐ │ context window keeps every prior tool call + result │ │ → it ALREADY HAS turn-2's grep output at turn 5 │ └───────────────┬───────────────────────────────────────────────┘ │ semfs grep "q" Request::Search { query, filepath } ▼ ↑ no session / offset / "seen" field ┌─ semfs DAEMON ────────────────────────────────────────────────┐ │ the PROCESS is long-lived & STATEFUL: owns index, mount, │ │ queue (state: Arc<IpcState>) … │ │ …but RETRIEVAL is a PURE FUNCTION: query → top-N → render │ │ it has ZERO memory of what it returned last call ◀── the gap │ └────────────────────────────────────────────────────────────────┘

So there are two stores and they don't talk: the daemon process is infrastructurally stateful (persists across calls per tag), but retrieval is stateless — Request::Search { query, filepath } has no session/offset/seen field (confirmed in ipc.rs). The agent keeps every prior result; the daemon keeps none.

"For p2b" specifically: p2b adds zero state. It's a pure prompt string (WB_TURNBRAKE) — a behavioral plea layered on the same stateless-retrieval / stateful-context split. That's why it's non-deterministic, and why we watched it get disobeyed (followed=False on 55/A; 9 calls under a "~5" budget on 53).

The trace that proves it — p2b, case 53 (verbatim)

1. semfs grep "interaction document" broad 2. semfs grep "interaction 6" ┐ 3. semfs grep "interaction 8" │ 4 rephrases of #1 — stateless daemon 4. semfs grep "interaction 13" │ re-sends the SAME top docs, full text 5. semfs grep "interaction 10" ┘ 6. find . -maxdepth 3 -type f | … ┐ 7. find -iname '*interaction*6*' … │ filesystem fallback 8. find | grep -Ei 'interaction|…' │ (retrieval miss → ranking) 9. find /Downloads /Desktop /ws … ┘ 10. cat > model_output/interaction_document_6.txt finally writes 9 discovery calls under a "~5 call" prompt → 0/22

⚠ "Didn't we already add this?" — No. We added a HINT, not dedup. agent_hint.rs:200-210 injects an AGENTS.md line: "Do NOT repeat a search you already ran with reworded queries — the files it returned are already in your context." The deterministic fix (no prompt): session-aware dedup

This is the open grep-stateless-context-dedup ticket — make the daemon remember what it just handed out:

  • Daemon keeps a bounded per-session ring of (query_embedding, returned_filepaths, turn).
  • On a ≥~0.9-similar follow-up query: don't re-send already-returned excerpts; emit one cheap label and return content only for new files.
  • -n paginates by offset (next results), not re-list from the top.
TODAY (stateless) grep "interaction document" → 4 docs, full text turn 2 grep "interaction 6" → SAME 4 docs, full text — re-paid turn 3 WITH SESSION DEDUP grep "interaction document" → 4 docs, full text turn 2 grep "interaction 6" → # already in context (turn 2): # interaction_document_6.txt — not resending # NEW: (none) turn 3 ← ~1 cheap line

Why this beats p2b: it doesn't ask the agent to behave — it physically stops re-sending bytes, so it works even when the agent disobeys (which it does). It lives in the long-lived daemon and degrades gracefully to today's stateless behavior on the mountless/Modal path (no session there). Caveat: it's a daemon code change → same Modal rebuild + E2B re-bake as offload.

Architecture: the cross-turn dedup mechanism

Three facts from the code decide the whole design: (1) the client (grep) is a fresh process every call — it cannot hold state; (2) the daemon is long-lived per mount/tag and already owns the index; (3) the daemon's search path (sqlite_vec.rs) already reconstructs whole-doc for the top-N and already reads SEMFS_RESULT_LIMIT. So the session memory must live in the daemon, and it seams in exactly where content is reconstructed.

① Where it lives — a small SessionMemory on IpcState, wrapping index.search:

one agent session = one mounted daemon (per tag) ┌──────────────────────────────────────────────────┐ │ semfs DAEMON (long-lived; owns index + state) │ ┌─────────┐ IPC │ Request::Search { query, filepath } │ │ grep │ (unix sock)│ │ │ │ fresh │────────────▶│ ▼ │ │ process │ │ ┌─────────────────┐ embed → RRF → rerank │ │ each │ │ │ index.search() │ │ │ call │◀────────────│ └────────┬────────┘ │ │ STATE- │ SearchHits │ ▼ │ │ LESS │ (+ seen │ partition top-N ──────▶ ┌────────────────────┐NEW│ └─────────┘ markers) │ seen? → marker only │ SESSION MEMORY │ │ │ new? → reconstruct + │ turn: u64 │ │ │ record ───────▶│ sent: ring of │ │ │ │ (path, first_turn)│ │ │ │ window W = 5 turns │ │ │ └────────────────────┘ │ └──────────────────────────────────────────────────┘ Modal / mountless path: NO daemon → NO session memory → stateless (today's behavior)

② The state — bounded, cheap, lives behind one mutex (greps are sequential per agent):

// on IpcState:   recent: Mutex<SessionMemory>
struct SessionMemory {
    turn: u64,                          // monotonic — the Nth search this session
    sent: VecDeque<(PathBuf, u64)>,     // (path, turn first sent w/ content); bounded ring
}

// hit gains ONE optional field — serde default None ⇒ wire-compatible
struct Hit { path, score, /* … */ seen_at_turn: Option<u64> }

③ The decision — wraps the existing search; the only behavior change is "skip reconstruction + mark" for files already sent inside the window:

// inside the daemon's Request::Search handler
let turn = mem.next_turn();                       // ++ per session
let mut hits = index.search(&query, filepath)?;   // UNCHANGED retrieval
for h in hits.iter_mut().take(result_limit()) {   // only the top-N get whole-doc today
    match mem.first_sent(&h.path, WINDOW) {
        Some(t) => h.seen_at_turn = Some(t),       // already in agent's context → pointer only
        None    => { reconstruct_whole_doc(h);     // new file → content exactly as today
                     mem.record(&h.path, turn); }
    }
}
mem.evict_before(turn - WINDOW);                   // keep memory + correctness bounded

The client change is tiny and additive: when a hit carries seen_at_turn = Some(n), grep.rs prints # already in your context (turn n): <path> — not resending instead of the excerpt. Everything else (caps, rendering) is unchanged.

④ Sequence — the case-53 spiral, fixed:

turn 2 grep "interaction document" search → {doc6, doc8, doc10, doc13} none in `sent` → reconstruct all, record {doc6..13 @ turn 2} return: full content (paid once) turn 3 grep "interaction 6" search → {doc6, …} doc6 ∈ `sent` (turn 2, within W) → seen_at_turn = 2 return: "# already in context (turn 2): doc6 — not resending" + content for any genuinely NEW hit only ← ~1 cheap line, not a re-dump

⑤ Guards & edges (the parts that keep it honest):

ConcernHandling
Content aged out of agent's context? Window W (default 5 turns): only suppress recent returns; older files are re-sendable. Bounds memory and correctness.
Different search, same file — over-suppress? Core is file-level (if it's in context, it's in context regardless of query). Optional refinement: gate on query-sim ≥ 0.9 to only dedup obvious re-searches — conservative, matches the ticket.
-n "give me the next ones" Track per-query hand-out count → advance an offset instead of re-listing from the top (the ticket's second symptom).
Mountless / Modal No daemon ⇒ no session ⇒ falls back to exactly today's stateless behavior. Zero risk to that path.
Concurrency One Mutex<SessionMemory>; per-agent greps are sequential, so contention is nil.

Why this is the deterministic fix: the suppression happens in the daemon before content is ever serialized — the agent cannot "ignore" it the way it ignores the hint/p2b, because the bytes are simply never sent. It reuses the existing search path untouched; the entire mechanism is one struct + one optional hit field + a partition loop. Same sqlite_vec.rs/daemon recompile as N4 — one rebuild ships both.

What N4 (offload) does

N4 = the offload lever (the 4th novel-idea lever from the optimization pass; a.k.a. "row-aligned offload"). What it does: today a grep hit reconstructs and inlines the whole-document text for each top-N file (sqlite_vec.rs:62). N4 instead returns a pointer (path + line range) + a tight snippet, and lets the agent read the rest on demand — only if it actually needs more.

TODAY (whole-doc inline) grep "q4 revenue" → hit: financials.xlsx <~12,000 chars of the WHOLE sheet inlined> ← re-paid EVERY later turn WITH N4 OFFLOAD (pointer + snippet, read-on-demand) grep "q4 revenue" → hit: financials.xlsx:88–104 Row 92 Q4 revenue …… 4.18M ← ~300 chars: just the matched window (read more: sed -n '1,200p' financials.xlsx) ← agent pulls the rest ONLY if it needs it

Why it cuts tokens: under cached_input=0 every inlined char is re-paid on every subsequent turn (total ≈ turns × per-turn-context). N4 shrinks per-turn-context at the source. The whole-doc reconstruction is hard-coded at sqlite_vec.rs:62 with no env switch — which is exactly why N4 needs the Modal recompile (it can't be swept like h1/h2/h3).

N4 (offload) vs dedup — different halves of the same waste. Offload makes each return smaller (don't inline the whole doc); dedup stops repeated returns across turns (don't re-send what the agent already has). Both are sqlite_vec.rs/daemon code, so one rebuild ships both.
