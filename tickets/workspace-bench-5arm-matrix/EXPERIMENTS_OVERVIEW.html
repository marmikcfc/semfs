What were we testing, and did it work?

One question drove the whole campaign: can semfs (semantic filesystem) make a coding agent cheaper or more accurate than a plain agent reading files directly?

The one-paragraph verdict

After fixing the infrastructure that made semfs look terrible, the local semfs arm reaches accuracy parity or better with plain on every case. But it does not reliably save tokens. It wins tokens only on a narrow case shape — single-answer questions whose answer file is not named in the task, where one capped semantic search replaces 4–8 blind find/grep probes. On broad workspace tasks the agent over-explores and semfs costs 1.4×–3.6× more. Conclusion: we were benchmarking on the wrong arena — on a small, well-named corpus, plain grep is already near-optimal, so there is little for semfs to win.

The finding, explained 5 ways

Click a level — same finding, from a 10-year-old to an engineer.

Why tokens cost what they cost (the physics)

Every result only makes sense once you see the cost equation. The agent is single-turn with cached_input=0: every tool-call turn re-sends the entire conversation so far. So the bill is not "how much did we read" — it's "how much did we read × how many turns remained after we read it."

TotalTokens ≈ T·S + Σ (T−s)·(oₛ + aₛ) + G T = number of tool-call turns S = static scaffold re-paid EVERY turn (system + task prompt + injected hint) oₛ = model-visible output of the tool call at turn s (≠ what the tool emitted!) s = the turn a big read happens — EARLY reads are re-paid more times G = generated/output tokens (small on WB: ~1–3K)

Law 1 · turns multiply scaffold

Every avoided turn saves a whole S re-read. Turn count T is the master dial.

Law 2 · early reads cost most

A read at turn 1 is re-paid (T−1) times. The hint commanded the 35 KB KG read at turn 1 — the worst possible slot.

Law 3 · output is a rounding error

Output G ≈ 1–3K vs 78–138K input. Input is ~96% of the bill — so output-compression (caveman) was de-prioritised.

The hidden truncation layer (changes how we read every number)

codex clips every tool output to ~10 KiB or 256 lines, head+tail, before the model sees it. A cloud run emitted 830K chars but the model was billed as if it saw ~10 KiB. So capping semfs grep at 6 KB isn't mainly "fewer bytes" — it's choosing which bytes survive instead of letting codex blindly cut the middle (possibly dropping the answer row). Put the answer first; the head survives.

★ The best token-saving config

Across the whole campaign, this is the configuration that reaches accuracy parity-or-better and saves tokens where any config can. It is the shipped default ("scout stack"). The honest caveat: it only wins tokens situationally.

SEMFS_SEARCH_ONLY=off # MANDATORY — "on" → catastrophic flail (H2). Agent must be able to fall back to reading files. SEMFS_GREP_RESULT_CAP=6144 # cap per-hit render to 6 KB — sits inside codex's ~10 KiB clip; WE choose which bytes survive SEMFS_GREP_TOTAL_CAP=10240 # global render budget — stops 5×6KB hits summing past the clip SEMFS_RESULT_LIMIT=5 # ranked hits returned SEMFS_KG=0 SEMFS_GRAPH_FS=0 # KG / by-topic surfaces are net-negative for codex-class agents (H5, O7) SEMFS_REWRITE=1 # cross-lingual query rewrite (zh task ↔ en files) — proven rank fix #417→#1 hint = v4.1 # "facts + costs only" — zero KG-first, zero honesty coaching (the bitter lesson)

✓ Where it saves tokens at no accuracy cost

Narrow, single-answer QA where the answer file is not named in the task → one capped semantic search replaces 4–8 blind discovery probes.

  • E8 case 175 — −7% tokens, accuracy ↑ (.083→.305)
  • E8 case 44 — −10% tokens, accuracy ↑
  • E11 e11-001 — −28% tokens, accuracy within tolerance
  • EC2 case 289 best run — 76.8K vs plain 79K (first-ever sub-plain)

✗ Where it costs more (the majority)

Broad workspace tasks where the agent explores widely, or tiny-file corpora where scanning is already cheap.

  • E8 case 289 — 3.6× tokens (390K vs 107K), accuracy ↑ only
  • E8 case 95 — 2.6× tokens, both score 0
  • E8 case 15 — 1.4× tokens, parity
  • E11 e11-002 — tied (+0.3%), files too small to stress discovery

The two levers that actually moved the needle

  1. The grep-cap code fix. semfs grep returned uncapped ~300K-char blobs; the cap knobs were inert on the CLI path until patched. Capping to 6 KB moved case 289 from 111–145K → a 77–98K band.
  2. Killing the KG-first hint. The baked AGENTS.md said "read kg/KNOWLEDGE_GRAPH.md FIRST" → 35.7K KG read + 22.4K os.walk = 58K wasted/run. Hint v4.1 (facts + costs only) removed it.

Residual blocker to a consistent win: run-to-run call-count variance (5 vs 16 calls) from the agent's own os.walk crawl. The fix is a code feature that was never built — E14 semfs slice.

Cross-cutting discovery — the benchmark bug

🔴 Case 95 is a judge filename lottery

The agent-visible task names no output file; the expected filename lives only in judge-side fields. Agents rationally name the file after the task's words — then a lenient judge grades content (12/12), a strict one zeroes on the name (0/12). Proven: two identical runs (same config, same deliverable filename) scored 0/12 and 12/12. 34/100 WB cases have this shape. Every case-95 score carries an asterisk; only the token/call axis is trustworthy there.

E14 · semfs slice tool highest-value code

What: semfs grep stores the full ranked result server-side and returns ≤1 KB + a result-id; a new semfs slice <id> --file N --range a:b retrieves precise pieces on demand.

Why it matters: this is the missing piece that would kill the run-to-run variance (the agent's os.walk crawl). Target flow: search(1KB) → slice(2KB) → write ≈ 25–30K tokens — the first floor-adjacent run.

Parked because: it's the only real code investment on the list; was gated on E9 signalling that small-payload delivery is safe (it did).

What the result means for us — criterion: n≥3 on case 289, mean tokens vs plain's ~79K at accuracy ≥ plain−1
✓ PASSa consistent 25–30K run at parity → this is the both-axes win we never got. semfs has a real, repeatable token edge via the artifact pattern (search→slice→write). We build & ship slice, and the product story finally leads with tokens.
✗ FAILtokens don't reach the floor, or accuracy drops → delivery optimization has hit its ceiling for codex. Stop chasing token wins through retrieval/delivery entirely; the remaining value of semfs is accuracy/acquisition, not cost.

E7c · KG-as-≤4 KB digest arm cheap

What: replace the 61 KB knowledge-graph file with a ≤4 KB digest (topic→dir table + inaccessible-files list).

Tests: O7's escape hatch — can KG be rehabilitated as a tiny digest rather than killed outright? Would beat nokg on neither or both axes.

Parked because: KG-off seed builds are impractically slow on the 16 GB box; now unblocked by the Modal SEMFS_INDEX_ONLY path.

What the result means for us — criterion: digest arm vs nokg on tokens & accuracy
✓ PASSdigest ≥ nokg on either axis → KG isn't dead, it was just too big. The 61 KB graph was the problem, not the idea. We ship the ≤4 KB digest as the orientation surface and rehabilitate a feature we'd written off.
✗ FAILdigest ≤ nokg on both → KG is net-negative even when tiny. Confirms O7 fully: kill KG/by-topic as an agent-facing surface for codex-class agents, end of story. Removes a whole feature branch from the roadmap.

E13 · ≤1 KB workspace map half-day

What: inject a ~1 KB map (top-level dirs, file-type census, "where things live") into the hint.

Tests: O2's alternative — if hints alone underperform, does a structural map cut discovery probes (4–8 → 1–2)? It would help both arms (retrieval-free).

Parked because: lower priority than the hint+cap levers that were already winning.

What the result means for us — criterion: discovery-probe count (find/ls/grep) with vs without the map
✓ PASSprobes drop 4–8 → 1–2 → a cheap, retrieval-free token lever. Ship the map in the hint. Caveat: it helps the plain arm too, so it's a general agent-scaffolding win, not a semfs differentiator — good for the product, neutral for the "semfs vs plain" thesis.
✗ FAILprobe count unchanged → codex doesn't steer on structural maps. Combined with the hint findings, this says behaviour isn't instruction-driven for this agent — drop maps, and stop expecting prompt/scaffold text to change exploration.

Benchmark bug · fix the judge filename lottery blocks all 95-class scores

Fix: either (a) the task states the output filename, or (b) the rubric accepts task-consistent names + matches on content with n≥2 judge runs. Until then, 34/100 WB cases produce coin-flip scores.

This is a measurement-validity fix, not a hypothesis — so the "result" is about trust, not pass/fail
✓ FIXEDcase-95-class scores become deterministic across judge runs → 34/100 WB cases become trustworthy. We can finally quote synthesis-case accuracy and re-read every asterisked result in this campaign (incl. the e9d compression confound).
✗ IF SKIPPEDevery synthesis-case accuracy number stays a coin flip → we keep mis-attributing judge noise to retrieval/compression/hint changes (exactly what happened in E9(d)). Any future accuracy claim on these cases is unsafe to quote.

Next steps — in priority order

  1. Decide the arena (strategic fork). E8 failed its kill condition and E11 didn't discriminate. The pre-committed branch says: stop optimising WB-chanpin tokens and re-aim at discovery-stressed + cross-lingual corpora. Decision needed: redesign the discovery corpus (E12-new), or pivot to the multi-benchmark adapter (xAFS / terminal-bench)?
  2. Build semfs slice (E14). The only lever with a credible path to a consistent floor-adjacent run (25–30K). Everything else is ±parity.
  3. Fix the judge filename lottery before quoting any synthesis-case accuracy again. It silently corrupts 34/100 cases.
  4. Run E7c (KG-digest) — now cheap via the Modal SEMFS_INDEX_ONLY path; settles whether KG dies or shrinks.
  5. If accuracy is the goal, not tokens: semfs already wins — nokg accuracy beat plain in 4/5 E8 cases. Lead the product story with acquisition (format-trap −80%) + cross-lingual (#417→#1), not raw token counts on grep-friendly ground.

The honest bottom line

There is a best token-saving config (the scout stack, already shipped), and it never hurts accuracy — but it only saves tokens on unnamed-answer single-file QA. On the benchmark we chose, plain grep is near-optimal, so the ceiling for a token win is low by construction. The next real result requires either a harder arena or the semfs slice code feature — not another knob.
