1. The one-sentence story

ELI5. We have a tool (semfs) that lets an AI "search by meaning" instead of guessing filenames. A benchmark said the tool made the AI worse. But the tool's engine was actually broken during the test — like blaming a race car for losing when its fuel line was cut. Fix the fuel line, and it keeps up.
Expert. The 5-arm matrix attributed a 3–14%-accuracy / 2–5×-token deficit to semfs retrieval. A trace audit + clean-infra reruns show 13/15 local runs hit hard infra faults (malformed DB, 50s search timeouts → 0-result cloud fallback). With a verified-clean seed + a health-gated driver, the local arm recovers to plain-competitive accuracy. The residual loss is on tokens, and it traces to one product mis-feature, not retrieval.

3. Four independent infra bugs — each could cause the "loss"

F1 Disk-full corruption The box ran at 93–95% disk. chunks has a last_accessed_at column → search performs writes. A write under ENOSPC tears a page → database disk image is malformed. The canonical seed's quick_check was always OK, so corruption was a runtime event in the working copy. Fix: disk guard + quick_check health-gate on every fresh copy.
F4 716 contamination sidecars (estimated as "6") The "clean" seed carried 716 <file>.semfs-error.txt twins (HTTP-402 "out of credits" from a failed cloud ingest) across the whole corpus + a baked model_output/. find/ls surfaced an error-twin for nearly every source file → the agent concludes the data is inaccessible and gives up or fabricates. Fix: rebuilt the seed via daemon FUSE rm (never raw SQL — desyncs ffts/vchunks).
F-cache Embedder pollutes the corpus → mount hang The adapter runs semfs mount with cwd = the workdir, so fastembed cached 898MB of model blobs into $WORKDIR/.fastembed_cache. An unscoped import then tries to re-index 898MB of model files → the mount hangs. Fix: strip it before each mount.
F-daemon Daemon survives unmount → vec0 corruption semfs unmount removes the mount but leaves daemon-inner holding the old DB mmap'd. The next run's cp overwrites the file underneath it → Error opening vector blob + constraint failures. Fix: kill daemon-inner before the fresh copy.

6. The residual token sink (and the one lever left)

Case 289's 111K breaks down as 35K reading kg/KNOWLEDGE_GRAPH.md + 22K os.walk + ~2K real work. The seed's AGENTS.md commands "read kg/KNOWLEDGE_GRAPH.md FIRST" → the agent obeys, greps scoped to kg/ (useless), then crawls. A clean simple grep returns the exact answer file as the top hit. The KG hint is the root cause — and it can't be removed from the existing seed (read-only /kg/; clearing graph_* triggers a hanging re-import).
The structural fix tried: a fresh seed with SEMFS_KG=off. Result: BLOCKED — fresh seeding is impractical on the 16GB box (~13 files/min → ~3h; half-indexed 259/2212 files, missing 289's answer file).

7. The deeper token bug — and why no config can win

A controlled A/B on the SAME query proved the documented delivery knobs are inert on the agent's semfs grep CLI:
semfs grep configoutput chars
CAPPED — RESULT_LIMIT=3, DOC_CAP=2048, snippet372,321
DEFAULT316,425
A broad query returns ~300K uncapped chars regardless of the caps — this, not just the KG, is why every semfs arm cost 2–5× plain's tokens. A clean win is blocked at the code layer (uncapped grep + a compiled-in hint), not the config layer.
Product fixes to make local semfs win: (1) make semfs grep honor RESULT_LIMIT/DOC_RETURN_CAP/RETURN_MODE on the CLI path; (2) rewrite the injected hint to "grep → read the single top hit → don't crawl → skip kg/"; (3) ship a pre-built KG-off seed.

8. The fix — implemented, and the first sub-plain run CODE FIX

Patched crates/semfs/src/cmd/grep.rs to cap the per-hit rendered text (new knob SEMFS_GREP_RESULT_CAP, default 6 KB) at all three render sites, with an honest TRUNCATED marker. Built incrementally (52 s) + deployed.
