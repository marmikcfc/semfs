2 · The physics — drag turns, watch the bill explode

Because cached_input=0, every turn re-reads the entire conversation so far. A tool output at turn s is paid again on every later turn — so cost grows quadratically in turn-count T, not linearly. This one fact explains the whole dataset.

7 · Deep dive — the floor lever, done the filesystem-consistent way

The 80% lever, grounded in real code paths (grep.rs, init.rs, the daemon IPC). Key correction: the original "semfs slice" idea invents a verb — but the filesystem already has the verb. Toggle the two worlds below and watch the running bill.

The key realization (from reading the code)

The daemon already computes the full ranked result set — every hit's filepath, matched chunk, similarity, and the file content behind it (Vec<SearchHit>, backend/mod.rs:21). Then the render loop throws ~90% of it away: grep_result_cap() trims each hit to 6KB and grep_total_cap() caps the whole render at 10KB (grep.rs:420–478). The expensive thing already exists in memory — today we delete it. slice keeps it behind a tiny pointer.

Wait — is slice even a filesystem command? (it isn't)

grep is consistent because it's literally a shadow of real grep: semfs init installs a shell wrapper where grep "natural language" → semantic search and grep -flag falls through to the real grep (init.rs SHELL_WRAPPER). semfs's naming rule is "overload the command the agent already reaches for" — not invent verbs. slice breaks that: there's no Unix slice, so the agent must learn it from the hint — and hint compliance is ~2/3, which is E14's own kill condition. slice's job is already a filesystem primitive the agent has and uses:

slice's jobthe filesystem command for it
exact line range a–bsed -n '40,60p' FILE · head/tail
filter lines by termgrep TERM FILE (real grep, with flags)
the whole readable formcat FILE

And in the local/mount arm the content is already a real file: the .extracted.md sibling (format-trap fix). The v4_i2 trace shows the agent already slicing with python Path reads unprompted — it reaches for native slicing, not a learned verb.

The consistent redesign — cite the path, let the FS slice
$ grep "top10 product transaction amount conversion rate" .
# ≤1KB back — top excerpt + ranked paths, each citing the readable sibling + a line hint
  ▸ hit 1  product_data/..._work_plan.xlsx.extracted.md   (sim 0.71)
       挑选爆款 | top10 | 成交金额 | 转化率 …   (answer rows ~40–60)
  ▸ hit 2  product_data/sales_summary.xlsx.extracted.md   (sim 0.63)

$ sed -n '40,60p' product_data/..._work_plan.xlsx.extracted.md
# exact verbatim rows — a command the agent already has, zero new verbs, never compressed

Same 3-turn floor (grep → sed → write), but nothing new to learn, so the ~2/3-compliance risk evaporates. The only real work: make grep return small + cite the exact sibling path and a line hint.

Why this is THE 80% lever (not the others)

Sections 2 & 5 proved the bill is turn-count, and the fat turns come from the agent's verify instinct having no cheap outlet — when it distrusts a 6KB excerpt it re-greps (another 10KB dump, re-paid every later turn) or crawls/reads whole files. slice gives that instinct a 1-turn, ~1KB outlet: "give me rows 40–60 of hit 2, verbatim." The floor flow grep(1KB) → slice(2KB) → write = 3 turns ≈ 21–30K tokens. That floor already happened 3 times by luck (21,473 / 21,743 / 20,754) — slice turns luck into the default. Kill condition: if the agent ignores slice in ≥2/3 runs, it's a hint/UX problem, not an architecture one.
